## Top-level fields

## configs

ConfigMaps and Secrets managed by the chart. Each key becomes the
Kubernetes resource name (prefixed with the release name); type
selects ConfigMap vs Secret.

**When to use:** Use for in-chart ConfigMaps/Secrets that are mounted by chart
deployments. For externally-managed secrets, prefer the secretRefs
section with valueFrom references.

**Common mistakes:**
- Putting binary content in data — base64 it into binaryData instead.
- Forgetting type: secret — defaults to configMap, which won't be base64-encoded.

**Example:**

```yaml
app-config:
  data:
    config.yaml: 'log_level: info

      '
  type: configMap
app-secrets:
  data:
    API_KEY: my-secret-key
  type: secret
```

## daemonSets

Node-level agents that run one pod per matching node (log shippers,
monitoring agents, CNI plugins, storage drivers).

**When to use:** Choose daemonSets when the workload must run on every node (or
every node matching a nodeSelector). For app workloads, prefer
deployments. For stateful per-pod identity, prefer statefulSets.

**Related fields:**
- `updateStrategy`
- `hostNetwork`
- `tolerations`

**Common mistakes:**
- Setting replicas (DaemonSet pod count is managed by node count).
- Setting hpa (DaemonSets cannot be horizontally autoscaled).
- Forgetting tolerations for node taints (control-plane, GPU nodes).
- Setting both updateStrategy.rollingUpdate.maxUnavailable=0 and maxSurge=0 — the rollout stalls; at least one must be > 0.

_Example use case:_ Fluent Bit log shipper with hostNetwork=true, tolerating all node
taints to run on every node.

**Example:**

```yaml
fluent-bit:
  containers:
    main:
      image: fluent/fluent-bit
      imageTag: '2.2'
  hostNetwork: true
  tolerations:
  - operator: Exists
```

## daemonSetsGeneral

Chart-wide defaults applied to every daemonSets.* entry. Supports a
narrow subset of DaemonSetSpec: nodeSelector, tolerations, affinity,
securityContext, priorityClassName, probes. The narrowness is
intentional — DaemonSets do not have replicas, HPA, rollout strategy,
or migration concepts, so the general block mirrors only the
scheduling/security/probe fields that are universally applicable.
Per-instance values override these.

**When to use:** Set platform-wide defaults here to avoid repeating boilerplate in
every daemonSets.<name>.

**Example:**

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
tolerations:
- operator: Exists
```

## deployments

Map of Deployment resources keyed by logical name. Each key becomes
the Kubernetes Deployment name (prefixed with the Helm release name).
Inherits defaults from deploymentsGeneral.

**When to use:** Use deployments for long-running stateless workloads. Each entry can
auto-create Service, Ingress, HPA, PDB, ServiceMonitor via flags.

**Related fields:**
- `/deploymentsGeneral`
- `/services`
- `/hpas`

**Example:**

```yaml
backend-api:
  containers:
    main:
      image: my-app
      imageTag: v1.0.0
  replicas: 3
```

## deploymentsGeneral

Global defaults applied to every entry under deployments. Same shape
as DeploymentSpec but without required fields. Per-deployment values
override the matching keys here (deep merge).

**When to use:** Set platform-wide defaults here to avoid repeating boilerplate in
every deployments.<name>.

**Example:**

```yaml
autoCreateSoftAntiAffinity: true
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
```

## hpas

Map of standalone HorizontalPodAutoscaler resources keyed by name.
Use this for HPAs that target resources not managed by this chart
(e.g. an externally-deployed Deployment).

**When to use:** Use for HPAs not tied to a chart deployment. For chart deployments
prefer the inline hpa: block under deployments.<name>.

**Example:**

```yaml
external-api:
  maxReplicas: 10
  metrics:
  - resource:
      name: cpu
      target:
        averageUtilization: 70
        type: Utilization
    type: Resource
  minReplicas: 2
  scaleTargetRef:
    kind: Deployment
    name: external-api
```

## httpRoutes

Map of standalone HTTPRoute (Gateway API) resources keyed by name.

**When to use:** Use for traffic-splitting (canary), header-based routing, or any
Gateway API feature not covered by simple Ingress.

**Example:**

```yaml
canary-route:
  hostnames:
  - subdomain: app
  rules:
  - backendRefs:
    - name: app-stable
      port: 8080
      weight: 90
    - name: app-canary
      port: 8080
      weight: 10
    matches:
    - path:
        type: PathPrefix
        value: /
```

## ingresses

Map of standalone Ingress resources keyed by logical name. Use this for
Ingresses that span multiple services (e.g. one shared external Ingress
with paths routing to different deployments).

**When to use:** Use when one Ingress must route to multiple backends. For 1-deployment
ingresses, prefer autoCreateIngress on the deployment.

**Example:**

```yaml
api-gateway:
  hosts:
  - paths:
    - path: /v1
      pathType: Prefix
      portName: http
      service: backend-v1
    - path: /v2
      pathType: Prefix
      portName: http
      service: backend-v2
    subdomain: api
```

## secretRefs

Map of reusable env-var groups keyed by name. Each group is a list of
env-var-to-Secret-key mappings. Containers reference a group by name
via containers.<name>.secretRefs: [groupName].

**When to use:** Use when the same Secret-derived env vars are needed by multiple
deployments or containers. Avoids repeating the same env mapping.

**Related fields:**
- `/$defs/ContainerSpec/properties/secretRefs`

**Example:**

```yaml
shared-secrets:
- name: API_KEY
  secretKeyRef:
    key: API_KEY
    name: app-secrets
- name: DB_PASSWORD
  secretKeyRef:
    key: password
    name: db-creds
```

## statefulSets

Workloads requiring stable network identity (DNS) and/or persistent
per-pod storage. Use for databases, queues, stateful caches.

**When to use:** Choose statefulSets when pods need stable hostnames, persistent
volumes that survive rescheduling, or strict ordering during
rolling updates. For stateless apps, prefer deployments.

**Related fields:**
- `serviceName`
- `volumeClaimTemplates`
- `autoCreateService`
- `podManagementPolicy`

**Common mistakes:**
- Omitting serviceName (required by Kubernetes).
- Using autoCreateService without headless mode (the chart defaults headless=true; only override if you have a specific reason).
- Reusing a name shared with a deployments key (component labels collide).
- Combining hpa with podManagementPolicy=OrderedReady — scale-down is sequential and slow, causing HPA oscillation. Prefer podManagementPolicy=Parallel when using HPA.
- Omitting metadata.name on a volumeClaimTemplates entry — Kubernetes requires it (it becomes the volume name and the PVC name prefix).

_Example use case:_ Single-replica Postgres with one persistent volume and an autoCreated
headless Service for stable DNS.

**Example:**

```yaml
postgres:
  autoCreateService: true
  containers:
    main:
      image: postgres
      imageTag: '16'
  serviceName: postgres
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes:
      - ReadWriteOnce
      resources:
        requests:
          storage: 20Gi
```

### ContainerSpec

A single container in a Deployment/Job/CronJob pod template.
The container name is the map key in containers.<name>.
Required: image (no registry prefix needed) and imageTag.

**When to use:** Use ContainerSpec anywhere a container template is needed:
deployments, jobs, cronJobs all share this schema.

**Common mistakes:**
- Forgetting imageTag — the chart requires it explicitly.
- Using port-number key in ports instead of a port-name key.

**Example:**

```yaml

  http:
    containerPort: 8080
    servicePort: 80

  limits:
    cpu: 500m
    memory: 512Mi
  requests:
    cpu: 100m
    memory: 128Mi
```

#### lifecycle

Container lifecycle hooks (passthrough to Kubernetes). postStart runs

**Example:**

### DeploymentDefaultsSpec

Defaults shape — identical to DeploymentSpec except no required fields.

#### initContainers

_Type:_ `object`

**When to use:** Use initContainers for one-shot setup work that must complete before the

_Type:_ `integer`

### k8s.io.api.apps.v1.DeploymentStrategy

Controls how the Deployment controller replaces old Pods with new ones
during a rollout. The `type` field is the discriminator and accepts
either `RollingUpdate` (gradual, the default) or `Recreate` (terminate
every old Pod before any new Pod starts). Only when `type: RollingUpdate`
may you populate `rollingUpdate` with `maxSurge` and `maxUnavailable`;
the chart's `StrategySpec` wrapper enforces this pairing at template
time so a stray `rollingUpdate` block with `type: Recreate` is caught
before it reaches the cluster.

**When to use:** Use `RollingUpdate` for stateless services that can tolerate two
versions running side-by-side; use `Recreate` only for singletons or
workloads that hold an exclusive resource lock. For StatefulSet-style
workloads prefer a real StatefulSet — Deployment with `Recreate` does
not give you stable identity.

**Related fields:**
- `strategy`
- `replicas`
- `minReadySeconds`

**Common mistakes:**
- Setting `maxSurge: 0` together with `maxUnavailable: 0` — the rollout has no room to move and stalls indefinitely; at least one must be > 0.
- Choosing `Recreate` for a workload that holds open connections — every Pod terminates before any new Pod is Ready, producing a full outage window equal to the startup time.
- Populating `rollingUpdate` while leaving `type: Recreate` — the chart's StrategySpec rejects this, but raw Kubernetes silently ignores `rollingUpdate` in that case, masking the misconfiguration.

**Example:**

```yaml
rollingUpdate:
  maxSurge: 25%
  maxUnavailable: 0
type: RollingUpdate
```

#### rollingUpdate

Rolling update config params. Present only if DeploymentStrategyType = RollingUpdate.

#### type

_Type:_ `string`

Type of deployment. Can be "Recreate" or "RollingUpdate". Default is RollingUpdate.

### k8s.io.api.apps.v1.RollingUpdateDeployment

Tuning knobs for the `RollingUpdate` Deployment strategy. `maxSurge`
caps how many Pods *above* the desired replica count the controller
may create while rolling out (extra capacity to absorb new traffic
before old Pods drain); `maxUnavailable` caps how many Pods *below*
desired may be unavailable at once. Both accept an absolute integer
or a percentage of `replicas` (e.g. `25%`). At least one must be
greater than zero — `maxSurge: 0` paired with `maxUnavailable: 0`
deadlocks the rollout. Only valid when `strategy.type: RollingUpdate`;
the chart's StrategySpec rejects this block under `Recreate`.

**When to use:** Tune when default rollout speed is wrong for the workload: raise
`maxSurge` to roll forward faster on large fleets; set
`maxUnavailable: 0` for zero-downtime rollouts. For singletons or
strict capacity caps prefer `strategy.type: Recreate` over forcing
these knobs to extremes.

**Related fields:**
- `strategy`
- `replicas`
- `minReadySeconds`

**Common mistakes:**
- Setting both `maxSurge: 0` and `maxUnavailable: 0` — the rollout has no room to make progress and stalls forever; at least one must be > 0.
- Using percentages on small replica counts without checking the rounding — `25%` of 2 replicas rounds to 1, which may not be what you wanted.
- Leaving `rollingUpdate` populated while flipping `strategy.type` to `Recreate` — the chart rejects this; raw Kubernetes silently ignores the block and masks the misconfiguration.

**Example:**

```yaml
maxSurge: 25%
maxUnavailable: 0
```

#### maxSurge

The maximum number of pods that can be scheduled above the desired number of pods. Value can be an absolute number (ex: 5) or a percentage of desired pods (ex: 10%). This can not be 0 if MaxUnavailable is 0. Absolute number is calculated from percentage by rounding up. Defaults to 25%. Example: when this is set to 30%, the new ReplicaSet can be scaled up immediately when the rolling update starts, such that the total number of old and new pods do not exceed 130% of desired pods. Once old pods have been killed, new ReplicaSet can be scaled up further, ensuring that total number of pods running at any time during the update is at most 130% of desired pods.

#### maxUnavailable

The maximum number of pods that can be unavailable during the update. Value can be an absolute number (ex: 5) or a percentage of desired pods (ex: 10%). Absolute number is calculated from percentage by rounding down. This can not be 0 if MaxSurge is 0. Defaults to 25%. Example: when this is set to 30%, the old ReplicaSet can be scaled down to 70% of desired pods immediately when the rolling update starts. Once new pods are ready, old ReplicaSet can be scaled down further, followed by scaling up the new ReplicaSet, ensuring that the total number of pods available at all times during the update is at least 70% of desired pods.

### k8s.io.api.autoscaling.v2.ExternalMetricSource

HPA metric source for signals that live outside the cluster — queue

(KEDA, cloud-provider metrics adapter); the bare cluster cannot

**When to use:** Use for autoscaling on signals that aren't pod or k8s-object metrics
— queue depth, request rate from a CDN, ML model load. Combines well

**Related fields:**
- `hpa.metrics`
- `target`

- Forgetting to install the external-metrics adapter — HPA shows `FailedGetExternalMetric` and scaling does not react.

### k8s.io.api.autoscaling.v2.HPAScalingPolicy

One rate-limit rule inside an HPAScalingRules block. `type` picks the
unit — `Pods` (absolute number of pods) or `Percent` (percent of
current replicas); `value` is the magnitude; `periodSeconds` is the
sliding window the policy applies over. Within an HPAScalingRules
block multiple policies are evaluated and the selectPolicy
(`Max`/`Min`/`Disabled`) picks which one wins for that direction.

**When to use:** Use `Percent` to cap scaling proportional to fleet size (good for
large deployments where doubling fast is fine but +1000 is not). Use
`Pods` to cap the absolute step size (good for small fleets where a
percent of 2 rounds awkwardly). Combine multiple policies inside one
HPAScalingRules and rely on `selectPolicy: Min` for conservative
behavior.

**Related fields:**
- `hpa.behavior`
- `scaleUp`
- `scaleDown`

**Common mistakes:**
- Setting `periodSeconds` longer than the HPA sync period (15s by default) without realizing the policy is evaluated only at sync ticks — short windows under 15s are effectively rounded up.
- Using `type: Percent` on a 1-replica Deployment — `Percent` of 1 with `value: 100` permits +1 pod per period, which is fine, but `value: 50` rounds to 0 and effectively disables scale-up.
- Defining a single policy and assuming `selectPolicy` doesn't matter — the default is `Max`, which picks the more aggressive policy across multiple entries; on `scaleDown` you usually want `Min` instead.

**Example:**

```yaml
periodSeconds: 60
type: Percent
value: 100
```

#### periodSeconds

_Type:_ `integer`

periodSeconds specifies the window of time for which the policy should hold true. PeriodSeconds must be greater than zero and less than or equal to 1800 (30 min).

#### type

_Type:_ `string`

type is used to specify the scaling policy.

#### value

_Type:_ `integer`

value contains the amount of change which is permitted by the policy. It must be greater than zero

### k8s.io.api.autoscaling.v2.HPAScalingRules

window (slowing aggressive drops); the default is 300s for scale-down
and 0s for scale-up. `policies` lists HPAScalingPolicy entries that
cap step magnitude; `selectPolicy` (`Max`, `Min`, `Disabled`) picks
which policy applies when multiple match. `Disabled` freezes scaling
in that direction entirely — useful as a kill switch for scale-down
during incidents.

**When to use:** Use under `behavior.scaleDown` to slow drops and avoid flapping
(longer stabilization window + `selectPolicy: Min`). Use under
`behavior.scaleUp` to allow fast bursts (`selectPolicy: Max`,
short window). Set `selectPolicy: Disabled` as an explicit "do not
scale this direction" — useful during planned events.

**Related fields:**
- `hpa.behavior`
- `scaleUp`
- `scaleDown`
- `policies`

**Common mistakes:**
- Setting a tiny `stabilizationWindowSeconds` on `scaleDown` (e.g. 30) — pods flap as load oscillates; the default 300s exists for good reason.
- Combining `selectPolicy: Max` with conservative-looking policies and being surprised by rapid scaling — `Max` picks the *most* aggressive applicable policy; switch to `Min` if you wanted the conservative one to bind.
- Forgetting `policies` is required — an empty HPAScalingRules block is rejected by the API.

**Example:**

```yaml
policies:
- periodSeconds: 60
  type: Percent
  value: 10
- periodSeconds: 60
  type: Pods
  value: 1
selectPolicy: Min
stabilizationWindowSeconds: 300
```

#### policies

_Type:_ `array`

policies is a list of potential scaling polices which can be used during scaling. If not set, use the default values: - For scale up: allow doubling the number of pods, or an absolute change of 4 pods in a 15s window. - For scale down: allow all pods to be removed in a 15s window.

#### selectPolicy

_Type:_ `string`

selectPolicy is used to specify which policy should be used. If not set, the default value Max is used.

#### stabilizationWindowSeconds

_Type:_ `integer`

stabilizationWindowSeconds is the number of seconds for which past recommendations should be considered while scaling up or scaling down. StabilizationWindowSeconds must be greater than or equal to zero and less than or equal to 3600 (one hour). If not set, use the default values: - For scale up: 0 (i.e. no stabilization is done). - For scale down: 300 (i.e. the stabilization window is 300 seconds long).

#### tolerance

tolerance is the tolerance on the ratio between the current and desired metric value under which no updates are made to the desired number of replicas (e.g. 0.01 for 1%). Must be greater than or equal to zero. If not set, the default cluster-wide tolerance is applied (by default 10%).

For example, if autoscaling is configured with a memory consumption target of 100Mi, and scale-down and scale-up tolerances of 5% and 1% respectively, scaling will be triggered when the actual consumption falls below 95Mi or exceeds 101Mi.

This is an beta field and requires the HPAConfigurableTolerance feature gate to be enabled.

### k8s.io.api.autoscaling.v2.HorizontalPodAutoscalerBehavior

Wrapper that bundles separate `scaleUp` and `scaleDown` HPAScalingRules
blocks. Sits under `hpa.behavior` and lets you shape each direction
independently — typically aggressive `scaleUp` to absorb traffic spikes
and conservative `scaleDown` (long stabilization window, `selectPolicy:
Min`) to avoid pod-flapping. Omit a direction's block to use the API
defaults (0s window + 100% step for scale-up; 300s window + 100% step
for scale-down).

**When to use:** Set this whenever you care about *how* the HPA reacts, not just *when*.
The default scale-up is already fast; tuning `scaleDown` is usually
where teams add value — a longer stabilization window prevents the
"scale up, scale down, scale up" cycle around midday/evening traffic
transitions.

**Related fields:**
- `hpa.behavior`
- `scaleUp`
- `scaleDown`

**Common mistakes:**
- Setting only one direction (e.g. only `scaleUp`) and assuming the other inherits — it does, but only from API defaults, which may not match your intent.
- Using `scaleDown: { selectPolicy: Disabled }` permanently — replicas can only grow, eventually consuming budget; this should be a temporary kill switch, not the steady state.
- Putting policies directly under `behavior` — the API expects `scaleUp` / `scaleDown` wrappers; flat layouts are rejected.

**Example:**

```yaml
scaleDown:
  policies:
  - periodSeconds: 120
    type: Pods
    value: 1
  stabilizationWindowSeconds: 600
scaleUp:
  policies:
  - periodSeconds: 30
    type: Percent
    value: 100
  stabilizationWindowSeconds: 0
```

#### scaleDown

scaleDown is scaling policy for scaling Down. If not set, the default value is to allow to scale down to minReplicas pods, with a 300 second stabilization window (i.e., the highest recommendation for the last 300sec is used).

#### scaleUp

scaleUp is scaling policy for scaling Up. If not set, the default value is the higher of:
  * increase no more than 4 pods per 60 seconds
  * double the number of pods per 60 seconds
No stabilization is used.

### k8s.io.api.autoscaling.v2.MetricIdentifier

Name + optional `selector` pair that picks out one metric from the
metrics adapter. The `name` is the metric's identifier as exposed by
the adapter (the custom-metrics or external-metrics API); `selector` is
a `LabelSelector` against the metric's labels and is needed only when
the same metric name exists with multiple dimensions (e.g. one
`queue_depth` per queue name).

**When to use:** Use a bare `name` when the metric is unique. Add a `selector` only
when you need to disambiguate which dimensional slice of the metric
to read — for example, picking one queue out of many.

**Related fields:**
- `metric`
- `pods`
- `object`
- `external`

**Common mistakes:**
- Misspelling `name:` — the metric is `<unknown>` and the HPA never reacts; the typo is silent at admission time.
- Using a `selector` with labels the metrics adapter doesn't expose on this metric — match returns zero series, the metric is unresolvable.
- Forgetting that some adapters require the metric name to be lowercase/normalized — e.g. Prometheus adapter strips characters and rewrites names by rule.

**Example:**

```yaml
name: http_requests_per_second
```

#### name

_Type:_ `string`

name is the name of the given metric

#### selector

selector is the string-encoded form of a standard kubernetes label selector for the given metric When set, it is passed as an additional parameter to the metrics server for more specific metrics scoping. When unset, just the metricName will be used to gather metrics.

### k8s.io.api.autoscaling.v2.MetricSpec

One entry in the HPA's `metrics[]` list, used by the chart inside
`hpa.metrics`. The `type` field is the discriminator and must match the
populated sibling block: `Resource` → `resource`, `ContainerResource` →
`containerResource`, `Pods` → `pods`, `Object` → `object`, `External`
→ `external`. Multiple MetricSpec entries are evaluated independently
and the HPA picks the highest desired replica count across them, so
adding a metric never lowers the floor — it can only raise the ceiling.

**When to use:** Use `Resource` for CPU/memory utilization (the common case); use
`Pods` for application-reported per-Pod metrics; use `Object` when the
signal lives on another Kubernetes object (e.g. Ingress RPS); use
`External` for off-cluster signals (queue depth, cloud metrics).
Prefer the v2 autoscaling API over the legacy v1 `targetCPUUtilization`
single-metric shortcut.

**Related fields:**
- `hpa.metrics`
- `hpa.minReplicas`
- `hpa.maxReplicas`
- `hpa.behavior`

**Common mistakes:**
- Setting `type: Resource` but omitting the `resource:` block (or filling `pods:` instead) — the HPA controller logs a validation error and the autoscaler stops reconciling that target.
- Mixing the legacy v1 `targetCPUUtilizationPercentage` shorthand with a v2 `metrics[]` list — the controller honors `metrics[]` and silently ignores the shorthand, which often looks like the threshold is wrong.
- Using `target.type: Utilization` against a container without a CPU `requests` value — utilization is computed as usage/request, so missing requests make the ratio undefined and the HPA reports `<unknown>`.

**Example:**

```yaml
resource:
  name: cpu
  target:
    averageUtilization: 70
    type: Utilization
type: Resource
```

#### containerResource

containerResource refers to a resource metric (such as those specified in requests and limits) known to Kubernetes describing a single container in each pod of the current scale target (e.g. CPU or memory). Such metrics are built in to Kubernetes, and have special scaling options on top of those available to normal per-pod metrics using the "pods" source.

#### external

external refers to a global metric that is not associated with any Kubernetes object. It allows autoscaling based on information coming from components running outside of cluster (for example length of queue in cloud messaging service, or QPS from loadbalancer running outside of cluster).

#### object

object refers to a metric describing a single kubernetes object (for example, hits-per-second on an Ingress object).

#### pods

pods refers to a metric describing each pod in the current scale target (for example, transactions-processed-per-second).  The values will be averaged together before being compared to the target value.

#### resource

resource refers to a resource metric (such as those specified in requests and limits) known to Kubernetes describing each pod in the current scale target (e.g. CPU or memory). Such metrics are built in to Kubernetes, and have special scaling options on top of those available to normal per-pod metrics using the "pods" source.

#### type

_Type:_ `string`

type is the type of metric source.  It should be one of "ContainerResource", "External", "Object", "Pods" or "Resource", each mapping to a matching field in the object.

### k8s.io.api.autoscaling.v2.MetricTarget

Threshold spec for an HPA metric source. `type` is the discriminator
and selects exactly one of `value` (absolute total), `averageValue`
(per-Pod total ÷ current replicas), or `averageUtilization` (percent
of `requests`, valid only for Resource/ContainerResource sources).
The HPA's desired replica count is computed so the observed metric
moves toward this target; the choice between `Value` and
`AverageValue` is the most common source of order-of-magnitude
scaling bugs.

**When to use:** Use `Utilization` for CPU/memory resource metrics (the common case)
to express targets as % of requests. Use `AverageValue` when the
metric is a per-Pod count (queue messages, RPS-per-Pod). Use plain
`Value` only when the metric is intrinsically singular (a total
cluster-wide rate that the HPA should divide across replicas).

**Related fields:**
- `target`
- `hpa.metrics`
- `resources`

**Common mistakes:**
- Mixing `type: Utilization` with a non-Resource metric source — utilization is only defined for built-in CPU/memory metrics; the API rejects it elsewhere.
- Setting `averageUtilization: 70` without `requests` on the target container — utilization is `usage/requests`; with no denominator the HPA reports `<unknown>`.
- Choosing `Value` when you meant `AverageValue` — `Value` is the total across all replicas; on a 10-replica fleet with `Value: 1000` the HPA tries to keep total at 1000, so each pod sees only ~100, which is rarely what you intended.

**Example:**

```yaml
averageUtilization: 70
type: Utilization
```

#### averageUtilization

_Type:_ `integer`

averageUtilization is the target value of the average of the resource metric across all relevant pods, represented as a percentage of the requested value of the resource for the pods. Currently only valid for Resource metric source type

#### averageValue

averageValue is the target value of the average of the metric across all relevant pods (as a quantity)

#### type

_Type:_ `string`

type represents whether the metric type is Utilization, Value, or AverageValue

#### value

value is the target value of the metric (as a quantity).

### k8s.io.api.autoscaling.v2.ObjectMetricSource

HPA metric source bound to a metric on a specific Kubernetes object —
typically RPS on an Ingress or connections-per-second on a Service.
`describedObject` (a CrossVersionObjectReference) points at the
object; `metric` (MetricIdentifier) names the metric; `target` is a
MetricTarget. Unlike `Pods`, the metric is read once for the object
(not averaged across replicas), so `target.type: Value` is the usual
choice and the HPA divides the target across the current replica
count internally.

**When to use:** Use when the signal lives on a k8s object (Ingress, Service,
GatewayAPI HTTPRoute) rather than on the Pods themselves. RPS
autoscaling against an Ingress is the canonical case — load
balancer metrics naturally aggregate at the Ingress, not the Pod.

**Related fields:**
- `object`
- `hpa.metrics`
- `describedObject`

**Common mistakes:**
- Setting `target.type: Utilization` — only valid for Resource/ContainerResource sources; the API rejects it here.
- Pointing `describedObject` at an object in a different namespace — the reference is namespace-local and the metric lookup silently fails.
- Using `AverageValue` for a metric the adapter exposes only as a total — the math the HPA does (target × replicas vs raw value) ends up off by a factor of replica count.

**Example:**

```yaml
describedObject:
  apiVersion: networking.k8s.io/v1
  kind: Ingress
  name: my-ingress
metric:
  name: requests_per_second
target:
  type: Value
  value: '1000'
```

#### describedObject

describedObject specifies the descriptions of a object,such as kind,name apiVersion

#### metric

metric identifies the target metric by name and selector

#### target

target specifies the target value for the given metric

### k8s.io.api.autoscaling.v2.PodsMetricSource

HPA metric source for per-Pod application metrics — anything an app
reports about itself, like in-flight requests, queue depth observed
per replica, or active goroutines. The metric (named by
MetricIdentifier) is read per matching Pod via the custom-metrics
API and averaged; `target.type` must be `AverageValue` (the only
value the API accepts here). Requires a custom-metrics adapter
(commonly the Prometheus adapter) running in the cluster.

**When to use:** Use when the autoscaling signal is a per-Pod application metric
reported via /metrics or similar. Pair with the Prometheus adapter
to expose Prometheus metrics through the custom-metrics API. For
built-in CPU/memory prefer `Resource`; for k8s-object metrics
prefer `Object`; for off-cluster signals prefer `External`.

**Related fields:**
- `hpa.metrics`
- `target`
- `metric`

**Common mistakes:**
- Setting `target.type` to anything other than `AverageValue` — the API rejects `Value` and `Utilization` for Pods metric sources.
- Forgetting to install/configure the custom-metrics adapter — HPA reports `FailedGetPodsMetric` and never scales.
- Naming the metric inconsistently with the adapter's exposed name — e.g. Prometheus adapter rewrites `http_requests_total` per its rules; the HPA name must match the *exposed* name, not the raw Prometheus name.

**Example:**

```yaml

  type: AverageValue
```

#### metric

metric identifies the target metric by name and selector

#### target

target specifies the target value for the given metric

### k8s.io.api.autoscaling.v2.ResourceMetricSource

The default HPA metric source for built-in Pod resources. `name`
picks `cpu` or `memory`; the value is *summed across all containers
in the Pod* and then averaged across Pods — this is the distinguishing
behavior from ContainerResource, which targets a single named
container. `target` is a MetricTarget; `Utilization` is the canonical
choice (percent of `requests`) but `AverageValue` is supported for
absolute thresholds. Requires the metrics-server (or an equivalent
metrics.k8s.io provider) running in the cluster.

**When to use:** Use this by default for CPU/memory autoscaling on single-container
workloads. For multi-container Pods where sidecar usage would
pollute the signal, switch to `ContainerResource`. For non-CPU/non-
memory signals, use `Pods` / `Object` / `External`.

**Related fields:**
- `resource`
- `hpa.metrics`
- `resources`

**Common mistakes:**
- Using `type: Utilization` while the target container has no CPU/memory `requests` — utilization is `usage/requests`; with no denominator the HPA shows `<unknown>` and never reacts.
- Picking `memory` without realizing Java/JVM apps rarely release heap — RSS plateaus and HPA never scales down; CPU is usually the better resource metric for JVM workloads.
- Setting `averageUtilization` over 100 — technically allowed (lets pods exceed requests on average), but a value over ~90 leaves no headroom for traffic spikes between HPA reconciliations.

**Example:**

```yaml
name: cpu
target:
  averageUtilization: 70
  type: Utilization
```

#### name

_Type:_ `string`

name is the name of the resource in question.

#### target

target specifies the target value for the given metric

### k8s.io.api.core.v1.Affinity

Kubernetes Affinity combines three independent rule sets — nodeAffinity,
podAffinity, and podAntiAffinity — that influence where the scheduler
can place pods. Use it for anything beyond simple key=value node label
matching (which `nodeSelector` covers more concisely). The chart auto-
injects soft pod anti-affinity per topology zone when
`autoCreateSoftAntiAffinity: true`; any podAntiAffinity you define
here wins over the auto-injected rule (merge semantics — your rules
are appended).

**When to use:** Use for scheduling preferences beyond key=value node-label matching.
Simple "place pod on node with label X" → prefer `nodeSelector`. Spread
pods across zones → prefer `topologySpreadConstraints` (modern, more
predictable than weighted podAntiAffinity).

**Related fields:**
- `nodeSelector`
- `topologySpreadConstraints`
- `autoCreateSoftAntiAffinity`

**Common mistakes:**
- Setting `requiredDuringSchedulingIgnoredDuringExecution` for soft preferences — that variant is HARD, pods stay Pending forever if no node matches. Use `preferredDuringSchedulingIgnoredDuringExecution` with a weight for soft.
- Mixing nodeAffinity and nodeSelector for the same constraint — they AND together at scheduling time; pods need to satisfy both, which is rarely what you meant.
- Forgetting `topologyKey` in podAffinityTerm — it has no default; the scheduler errors at admission time.

**Example:**

```yaml
nodeAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
    nodeSelectorTerms:
    - matchExpressions:
      - key: node-type
        operator: In
        values:
        - memory-optimized
```

#### nodeAffinity

Describes node affinity scheduling rules for the pod.

#### podAffinity

Describes pod affinity scheduling rules (e.g. co-locate this pod in the same node, zone, etc. as some other pod(s)).

#### podAntiAffinity

Describes pod anti-affinity scheduling rules (e.g. avoid putting this pod in the same node, zone, etc. as some other pod(s)).

### k8s.io.api.core.v1.CSIVolumeSource

### k8s.io.api.core.v1.Capabilities

Linux kernel capabilities the container gets in addition to (or
removed from) the runtime's default capability set. `add[]` grants
caps; `drop[]` removes them; both accept canonical names without the
`CAP_` prefix (e.g. `NET_BIND_SERVICE`, not `CAP_NET_BIND_SERVICE`).
The Pod Security Standard `restricted` profile requires `drop:
[ALL]` and forbids any `add` other than `NET_BIND_SERVICE`. Only
valid inside container `securityContext`, not Pod-level
`securityContext`. The `ALL` token in `drop` is the runtime's full
set; combining `drop: [ALL]` with explicit `add` is the standard
least-privilege pattern.

**When to use:** Always pair `drop: [ALL]` with whatever capabilities the workload
actually needs (most need none). The most common single
capability is `NET_BIND_SERVICE` for binding to ports below 1024
without root. For workloads that don't bind privileged ports,
`drop: [ALL]` alone meets `restricted` PSS.

**Related fields:**
- `securityContext`
- `allowPrivilegeEscalation`
- `runAsNonRoot`

**Common mistakes:**
- Using only `add` without `drop: [ALL]` — you keep the runtime's full default set and merely add to it, defeating least-privilege; almost always not what was intended.
- Including the `CAP_` prefix in capability names (`CAP_NET_BIND_SERVICE`) — Kubernetes rejects this format; names must be unprefixed.
- Granting capabilities like `SYS_ADMIN` instead of figuring out which specific capability is missing — `SYS_ADMIN` is broad enough that workloads often pass with it but the result is effectively as privileged as root.

**Example:**

```yaml
add:
- NET_BIND_SERVICE
drop:
- ALL
```

#### add

_Type:_ `array`

Added capabilities

#### drop

_Type:_ `array`

Removed capabilities

### k8s.io.api.core.v1.ConfigMapVolumeSource

`name` references the ConfigMap by name in the Pod's namespace.
Without `items` every key in the ConfigMap becomes a file at the
mount path; with `items` only the listed keys are projected, each
with an optional `path` (relative to the mount path) and `mode`

all files not given a per-item mode. `optional: true` lets the Pod

`optional: false` (the default) keeps the Pod Pending until it
exists. ConfigMap updates propagate to mounted files atomically

individual keys are mounted via `subPath` in the container.

**When to use:** Use for non-secret configuration files. For credentials use
`SecretVolumeSource`. For combining ConfigMap + Secret + downward

**Related fields:**
- `volumes`
- `volumeMounts`
- `configs`

**Common mistakes:**
- Using `subPath: <key>` in the matching `volumeMounts[]` to land the file as a single entry — Kubernetes uses non-atomic rename for subPath mounts, so ConfigMap updates *never* propagate; the file is frozen at Pod start.

- Setting `defaultMode` as decimal (e.g. `420` for 0644) and getting unexpected permissions — modes are octal; specify with leading zero or use the integer equivalent carefully.

**Example:**

```yaml
name: app-config
```

### k8s.io.api.core.v1.EmptyDirVolumeSource

Empty scratch volume created when the Pod is assigned to a node and
destroyed when the Pod is removed from that node — so it persists
across container restarts inside the same Pod, but *not* across
rescheduling. `medium: ""` (the default) backs it with the node's
default filesystem (typically a directory under
`/var/lib/kubelet/pods/`); `medium: Memory` backs it with tmpfs,
which is fast but counts against the Pod's memory limit and reduces
schedulable memory. `sizeLimit` caps total usage and kicks in at
eviction time — workloads exceeding the cap are evicted, not blocked
at write time.

**When to use:** Use for scratch space that lives with the Pod — caches, temporary
output, shared file between init and main containers. For data
that must survive Pod rescheduling use a PVC. For tmpfs-backed
scratch (faster, sized in RAM) set `medium: Memory`; remember the
memory counts against the container limit.

**Related fields:**
- `volumes`
- `volumeMounts`
- `readOnlyRootFilesystem`

**Common mistakes:**
- Using emptyDir for data that must survive Pod rescheduling — rescheduling to another node returns an empty directory. Use a PVC.
- Setting `medium: Memory` without raising the container `memory` limit — tmpfs writes count against the limit; the Pod OOM-kills as the volume fills.
- Forgetting `sizeLimit` on a shared write target — runaway writes can fill the node's ephemeral storage and impact other Pods on the same node.

**Example:**

```yaml
{}
```

#### medium

_Type:_ `string`

medium represents what type of storage medium should back this directory. The default is "" which means to use the node's default medium. Must be an empty string (default) or Memory. More info: https://kubernetes.io/docs/concepts/storage/volumes#emptydir

#### sizeLimit

sizeLimit is the total amount of local storage required for this EmptyDir volume. The size limit is also applicable for memory medium. The maximum usage on memory medium EmptyDir would be the minimum value between the SizeLimit specified here and the sum of memory limits of all containers in a pod. The default is nil which means that the limit is undefined. More info: https://kubernetes.io/docs/concepts/storage/volumes#emptydir

### k8s.io.api.core.v1.EphemeralVolumeSource

Inline PVC creation — the kubelet materializes a per-Pod
PersistentVolumeClaim from `volumeClaimTemplate` at Pod creation,
and deletes it when the Pod is deleted. The generated PVC's name is
`<pod-name>-<volume-name>` and it inherits the Pod's namespace.
Behaves like an EmptyDir bound to durable storage: data does *not*
survive Pod rescheduling (a new Pod gets a fresh PVC) but does
survive container restarts inside the same Pod. Requires a
dynamic provisioner (CSI driver with a StorageClass) for the
claim to bind; static PV provisioning is incompatible with
ephemeral volumes.

**When to use:** Use when each Pod replica needs its own durable scratch volume that
should disappear with the Pod — large workspace dirs, per-Pod
caches that benefit from real storage rather than tmpfs/emptyDir.
For shared durable storage across replicas, use a regular
PersistentVolumeClaim with `ReadWriteMany`. For shared scratch, use
`EmptyDirVolumeSource`.

**Related fields:**
- `persistentVolumeClaim`
- `volumes`
- `volumeClaimTemplate`

**Common mistakes:**
- Forgetting that the PVC is *deleted with the Pod* — assuming data persists across Deployment rollouts is wrong; rollouts replace Pods, which destroys ephemeral PVCs.
- Using `accessModes: [ReadWriteMany]` on a provider that doesn't support it — the PVC stays Pending and the Pod never starts.
- Setting `storageClassName: ""` (empty string) intending to use the cluster default — empty string means "no storage class" (use static binding), which usually breaks dynamic provisioning. Omit the field entirely to use the default class.

**Example:**

```yaml
volumeClaimTemplate:
  spec:
    accessModes:
    - ReadWriteOnce
    resources:
      requests:
        storage: 10Gi
    storageClassName: standard
```

#### volumeClaimTemplate

Will be used to create a stand-alone PVC to provision the volume. The pod in which this EphemeralVolumeSource is embedded will be the owner of the PVC, i.e. the PVC will be deleted together with the pod.  The name of the PVC will be `<pod name>-<volume name>` where `<volume name>` is the name from the `PodSpec.Volumes` array entry. Pod validation will reject the pod if the concatenated name is not valid for a PVC (for example, too long).

An existing PVC with that name that is not owned by the pod will *not* be used for the pod to avoid using an unrelated volume by mistake. Starting the pod is then blocked until the unrelated PVC is removed. If such a pre-created PVC is meant to be used by the pod, the PVC has to updated with an owner reference to the pod once the pod exists. Normally this should not be necessary, but it may be useful when manually reconstructing a broken cluster.

This field is read-only and no changes will be made by Kubernetes to the PVC after it has been created.

Required, must not be nil.

### k8s.io.api.core.v1.ExecAction

Probe / lifecycle-hook action that runs a command inside the container
using the kubelet's `exec` (essentially `kubectl exec` semantics).
`command` is the argv list — no shell, so `["sh", "-c", "..."]` is
needed for shell features. The command runs with the container's
working directory and user; exit code 0 is success, anything else
is failure. Exec probes are the most expensive probe type (fork +
process spawn per probe interval); for HTTP services prefer
`httpGet`, for raw TCP listeners prefer `tcpSocket`.

**When to use:** Use when the workload can't expose an HTTP or TCP health endpoint —
e.g. CLI tools, databases probed via their native client
(`pg_isready`, `redis-cli ping`), or workloads that signal health by
writing a sentinel file. For HTTP services use `httpGet` instead;
for TCP listeners use `tcpSocket`. Both are dramatically cheaper.

**Related fields:**
- `livenessProbe`
- `readinessProbe`
- `exec`

**Common mistakes:**
- Writing `command: ["pg_isready -U postgres"]` (one string) instead of `["pg_isready", "-U", "postgres"]` — the kubelet does no shell parsing; the single string is treated as the executable name and `exec` fails.
- Running expensive checks (full DB queries, full cache walks) on every probe — exec probes execute per interval; this becomes a measurable load source.
- Probing a process started via initContainer side effects — initContainers run before the main container, but exec probes only target the main container.

**Example:**

```yaml
command:
- cat
- /tmp/healthy
```

#### command

_Type:_ `array`

Command is the command line to execute inside the container, the working directory for the command  is root ('/') in the container's filesystem. The command is simply exec'd, it is not run inside a shell, so traditional shell instructions ('|', etc) won't work. To use a shell, you need to explicitly call out to that shell. Exit status of 0 is treated as live/healthy and non-zero is unhealthy.

### k8s.io.api.core.v1.HTTPGetAction

**Related fields:**

- `readinessProbe`
- `startupProbe`
- `httpGet`

**Common mistakes:**

- Mixing up readiness and liveness probes — readiness controls Service endpoints; liveness restarts containers. Pointing both at the same heavy path can cascade restarts during transient load.

### k8s.io.api.core.v1.Lifecycle

Per-container hooks the kubelet invokes around the container's lifetime.
`postStart` fires immediately after the container is created (no
guarantee it runs before the entrypoint, and a slow hook delays the
container reaching the Running state). `preStop` is by far the more
common use: it runs synchronously when a Pod is asked to terminate and
the container stays in Terminating until the hook returns or the Pod's
`terminationGracePeriodSeconds` elapses, whichever comes first — once
the grace period is exhausted the kubelet sends SIGKILL regardless.
Use preStop to drain in-flight requests, deregister from a service
registry, or sleep long enough for endpoints removal to propagate.

**When to use:** Use `preStop` whenever the workload accepts external traffic — the
`sleep N` pattern bridges the gap between SIGTERM and endpoint
removal. Use `postStart` only for genuinely fast initialization that
cannot live in an init container; init containers are the correct
tool for warm-up work that must complete before main traffic.

**Related fields:**
- `terminationGracePeriodSeconds`
- `readinessProbe`
- `initContainers`

**Common mistakes:**
- preStop sleep or work that exceeds `terminationGracePeriodSeconds` — the kubelet sends SIGKILL on expiry, the hook's effect is lost, and in-flight requests fail. Always keep preStop duration < grace period.
- Using `httpGet` for preStop against an endpoint the readiness probe has already taken out of rotation — the hook can still hit it locally, but pairing httpGet with a slow upstream chain can stall the entire grace period.
- Relying on `postStart` ordering vs the entrypoint — Kubernetes does not guarantee postStart runs before the container's main process, so the hook cannot safely assume the app is initialized.

**Example:**

```yaml
preStop:
  exec:
    command:
    - /bin/sh
    - -c
    - sleep 15
```

#### postStart

PostStart is called immediately after a container is created. If the handler fails, the container is terminated and restarted according to its restart policy. Other management of the container blocks until the hook completes. More info: https://kubernetes.io/docs/concepts/containers/container-lifecycle-hooks/#container-hooks

#### preStop

PreStop is called immediately before a container is terminated due to an API request or management event such as liveness/startup probe failure, preemption, resource contention, etc. The handler is not called if the container crashes or exits. The Pod's termination grace period countdown begins before the PreStop hook is executed. Regardless of the outcome of the handler, the container will eventually terminate within the Pod's termination grace period (unless delayed by finalizers). Other management of the container blocks until the hook completes or until the termination grace period is reached. More info: https://kubernetes.io/docs/concepts/containers/container-lifecycle-hooks/#container-hooks

#### stopSignal

_Type:_ `string`

StopSignal defines which signal will be sent to a container when it is being stopped. If not specified, the default is defined by the container runtime in use. StopSignal can only be set for Pods with a non-empty .spec.os.name

### k8s.io.api.core.v1.LifecycleHandler

Action shape for `postStart` and `preStop` lifecycle hooks. Populate
exactly one of `exec`, `httpGet`, or `sleep` (the latter added in
Kubernetes 1.29 as a simpler way to express the "wait N seconds"
pattern that used to require `exec: ["sleep", "N"]`). `tcpSocket`
is *not* available here — lifecycle hooks support a strict subset
of probe actions. Hook failures are logged but don't restart the
container directly; `preStop` blocking past
`terminationGracePeriodSeconds` causes SIGKILL.

**When to use:** Use `sleep` for the canonical "wait for endpoint removal to
propagate" preStop hook (Kubernetes 1.29+). On older clusters use
`exec: ["sleep", "15"]` instead. Use `httpGet` to ping a graceful-
shutdown endpoint inside the workload; use `exec` for non-HTTP
apps that drain via CLI.

**Related fields:**
- `lifecycle`
- `preStop`
- `postStart`

**Common mistakes:**
- Trying to use `tcpSocket` here — not supported in LifecycleHandler; only `exec`, `httpGet`, and `sleep` (1.29+) are valid.
- Letting `preStop` work exceed `terminationGracePeriodSeconds` — the kubelet SIGKILLs at expiry regardless of hook progress; tune grace period accordingly.
- Setting `postStart` to `httpGet` against the workload's own port and racing the entrypoint — there is no ordering guarantee between postStart and entrypoint; the GET may hit a not-yet-listening port.

**Example:**

```yaml
sleep:
  seconds: 15
```

#### exec

Exec specifies a command to execute in the container.

#### httpGet

HTTPGet specifies an HTTP GET request to perform.

#### sleep

Sleep represents a duration that the container should sleep.

#### tcpSocket

Deprecated. TCPSocket is NOT supported as a LifecycleHandler and kept for backward compatibility. There is no validation of this field and lifecycle hooks will fail at runtime when it is specified.

### k8s.io.api.core.v1.NodeAffinity

Scheduling rules expressed over node labels. Two independent fields:
`requiredDuringSchedulingIgnoredDuringExecution` (a hard NodeSelector
— pods stay Pending unless at least one term matches) and
`preferredDuringSchedulingIgnoredDuringExecution` (a list of weighted
preferences — the scheduler biases placement but does not block).
Both `IgnoredDuringExecution` suffixes mean Kubernetes does NOT
reschedule running pods when node labels change after placement;
there is no `RequiredDuringExecution` variant in mainline Kubernetes.
For simple key=value matching prefer `nodeSelector` for brevity.

**When to use:** Use `required...` when scheduling onto the wrong nodes is unsafe (GPU
workloads, license-constrained nodes, dedicated tiers). Use
`preferred...` for soft preferences with fallback. For the simple
`key=value` case `nodeSelector` is shorter and equivalent.

**Related fields:**
- `nodeSelector`
- `affinity`
- `topologySpreadConstraints`

**Common mistakes:**
- Using `required...` for what should be a preference — pods stay Pending forever if no node matches; switch to `preferred...` with a `weight`.
- Combining nodeSelector and a NodeAffinity term for the same constraint — they AND together at scheduling time; pods must satisfy both.
- Forgetting that `IgnoredDuringExecution` means running pods are not evicted when labels change — relabeling a node won't move existing workloads.

**Example:**

```yaml
requiredDuringSchedulingIgnoredDuringExecution:
  nodeSelectorTerms:
  - matchExpressions:
    - key: node-type
      operator: In
      values:
      - gpu
```

#### preferredDuringSchedulingIgnoredDuringExecution

_Type:_ `array`

The scheduler will prefer to schedule pods to nodes that satisfy the affinity expressions specified by this field, but it may choose a node that violates one or more of the expressions. The node that is most preferred is the one with the greatest sum of weights, i.e. for each node that meets all of the scheduling requirements (resource request, requiredDuringScheduling affinity expressions, etc.), compute a sum by iterating through the elements of this field and adding "weight" to the sum if the node matches the corresponding matchExpressions; the node(s) with the highest sum are the most preferred.

#### requiredDuringSchedulingIgnoredDuringExecution

If the affinity requirements specified by this field are not met at scheduling time, the pod will not be scheduled onto the node. If the affinity requirements specified by this field cease to be met at some point during pod execution (e.g. due to an update), the system may or may not try to eventually evict the pod from its node.

### k8s.io.api.core.v1.NodeSelector

Disjunctive (OR) collection of NodeSelectorTerms. Wraps the hard
`requiredDuringSchedulingIgnoredDuringExecution` rule in NodeAffinity:
a node satisfies the selector when *at least one* term matches. Each
term is itself a conjunction (AND) of `matchExpressions` and
optional `matchFields`. This is a different shape from the flat
`nodeSelector` map on PodSpec — that one is map-style key=value AND;
this one is list-style OR of ANDs and is much more expressive.

**When to use:** Use inside `nodeAffinity.requiredDuringSchedulingIgnoredDuringExecution`
whenever the hard rule can't be expressed as a single AND of
key=value pairs — e.g. "us-east-1a with m5.large OR us-east-1b
with m5.large". For pure AND of key=value, `nodeSelector` on the
PodSpec is shorter.

**Related fields:**
- `nodeAffinity`
- `nodeSelectorTerms`
- `matchExpressions`

**Common mistakes:**
- Confusing semantics: a single term with multiple `matchExpressions` is AND, but multiple terms in `nodeSelectorTerms` are OR. Beginners often flip this and produce a rule that's much narrower or broader than intended.
- Omitting `nodeSelectorTerms` entirely — an empty NodeSelector matches *no* nodes; the pod is Pending forever.
- Using `In: []` with an empty values list — rejected by the API, but with `NotIn: []` Kubernetes treats it as 'never matches' which is often surprising.

**Example:**

```yaml
nodeSelectorTerms:
- matchExpressions:
  - key: node-type
    operator: In
    values:
    - gpu
    - tpu
```

#### nodeSelectorTerms

_Type:_ `array`

Required. A list of node selector terms. The terms are ORed.

### k8s.io.api.core.v1.NodeSelectorRequirement

One match expression against a node label. `key` is the label name,
`operator` is one of `In`, `NotIn`, `Exists`, `DoesNotExist`, `Gt`,
`Lt`, and `values` is a list whose meaning depends on the operator:
with `In`/`NotIn` it must be non-empty; with `Exists`/`DoesNotExist`
it must be empty; with `Gt`/`Lt` it must be exactly one integer
string. `Gt`/`Lt` compare numeric labels only — labels are strings
in Kubernetes and the comparator parses them as integers, so
decimal/non-numeric values fail validation.

**When to use:** Use as a leaf inside a NodeSelectorTerm's `matchExpressions` (or
`matchFields`) — list as many requirements as needed; all of them
AND together within the term. For simple equality across the whole
PodSpec, prefer `nodeSelector` for brevity.

**Related fields:**
- `nodeSelectorTerms`
- `matchExpressions`
- `matchFields`

**Common mistakes:**
- Using `In` without a non-empty `values` list (or `Exists` with a `values` list) — the API validates this and rejects the pod.
- Using `Gt`/`Lt` against a label that isn't strictly integer-valued — comparison is numeric, non-numeric labels never satisfy the expression.
- Forgetting `NotIn` matches nodes that *also* lack the label — set theory: 'not in {x,y}' includes 'has no value at all'. If you want 'has the label but not these values', combine `Exists` AND `NotIn`.

**Example:**

```yaml
key: node-type
operator: In
values:
- gpu
- tpu
```

#### key

_Type:_ `string`

The label key that the selector applies to.

#### operator

_Type:_ `string`

Represents a key's relationship to a set of values. Valid operators are In, NotIn, Exists, DoesNotExist. Gt, and Lt.

#### values

_Type:_ `array`

An array of string values. If the operator is In or NotIn, the values array must be non-empty. If the operator is Exists or DoesNotExist, the values array must be empty. If the operator is Gt or Lt, the values array must have a single element, which will be interpreted as an integer. This array is replaced during a strategic merge patch.

### k8s.io.api.core.v1.NodeSelectorTerm

One AND-conjunction of node requirements. Pairs `matchExpressions`
(against node labels — the common case) with optional `matchFields`
(against node field paths like `metadata.name`). All entries in
both lists must hold for the term to match. Multiple terms inside
a NodeSelector are OR-combined, so the typical pattern is "many
small terms = OR alternatives" while "one term with many entries
= AND constraints".

**When to use:** Use as elements of `nodeSelectorTerms` or inside
`preferredDuringSchedulingIgnoredDuringExecution[].preference`.
Prefer multiple terms (OR semantics) over inflating one term with
conditional logic — the OR-of-ANDs shape is the standard mental
model for node selection.

**Related fields:**
- `nodeSelectorTerms`
- `matchExpressions`
- `matchFields`

**Common mistakes:**
- Mixing what should be OR alternatives into a single term's `matchExpressions` and ending up with a term that can never match (e.g. `zone In [a]` AND `zone In [b]`).
- Using `matchFields` for label data — `matchFields` targets API field paths, not labels; the lookup silently returns empty.
- Empty matchExpressions AND empty matchFields — the term has no requirements and is treated as 'match all', which is rarely the intent inside a hard rule.

**Example:**

```yaml
matchExpressions:
- key: node-type
  operator: In
  values:
  - memory-optimized
- key: topology.kubernetes.io/zone
  operator: In
  values:
  - us-east-1a
  - us-east-1b
```

#### matchExpressions

_Type:_ `array`

A list of node selector requirements by node's labels.

#### matchFields

_Type:_ `array`

A list of node selector requirements by node's fields.

### k8s.io.api.core.v1.PersistentVolumeClaimSpec

Desired state of a PersistentVolumeClaim. `accessModes` is the list of
mount modes the bound PV must support (`ReadWriteOnce`,
`ReadOnlyMany`, `ReadWriteMany`, `ReadWriteOncePod`); `resources`
is a VolumeResourceRequirements with `requests.storage` setting the
minimum size; `storageClassName` picks the StorageClass for dynamic
provisioning (omit for the cluster default, `""` explicit for static
binding only); `volumeMode` is `Filesystem` (default) or `Block`;
`dataSource`/`dataSourceRef` clones from a snapshot or PVC. After
creation, most fields are immutable — `resources.requests.storage`
is the only one that can be grown (and only if the StorageClass has
`allowVolumeExpansion: true`).

**When to use:** Use as the `spec` of a top-level PVC or inside an
`EphemeralVolumeSource.volumeClaimTemplate`. Always specify
`accessModes` (no sensible default) and `resources.requests.storage`
(required). Omit `storageClassName` to use the cluster default;
set it explicitly for tiered storage (e.g. `gp3` vs `io2`).

**Related fields:**
- `persistentVolumeClaims`
- `storageClassName`
- `accessModes`

**Common mistakes:**
- Setting `storageClassName: ""` (empty string) intending to use the cluster default — `""` explicitly means 'no class, static binding only'; omit the field entirely for default.
- Specifying `accessModes: [ReadWriteMany]` on a provider that doesn't support it (most block-storage providers don't) — the PVC stays Pending forever.
- Trying to shrink storage by re-applying with a smaller `requests.storage` — shrinking is not supported; the apply is rejected or silently ignored depending on driver.

**Example:**

```yaml
accessModes:
- ReadWriteOnce
resources:
  requests:
    storage: 10Gi
storageClassName: standard
```

#### accessModes

_Type:_ `array`

accessModes contains the desired access modes the volume should have. More info: https://kubernetes.io/docs/concepts/storage/persistent-volumes#access-modes-1

#### dataSource

dataSource field can be used to specify either: * An existing VolumeSnapshot object (snapshot.storage.k8s.io/VolumeSnapshot) * An existing PVC (PersistentVolumeClaim) If the provisioner or an external controller can support the specified data source, it will create a new volume based on the contents of the specified data source. When the AnyVolumeDataSource feature gate is enabled, dataSource contents will be copied to dataSourceRef, and dataSourceRef contents will be copied to dataSource when dataSourceRef.namespace is not specified. If the namespace is specified, then dataSourceRef will not be copied to dataSource.

#### dataSourceRef

dataSourceRef specifies the object from which to populate the volume with data, if a non-empty volume is desired. This may be any object from a non-empty API group (non core object) or a PersistentVolumeClaim object. When this field is specified, volume binding will only succeed if the type of the specified object matches some installed volume populator or dynamic provisioner. This field will replace the functionality of the dataSource field and as such if both fields are non-empty, they must have the same value. For backwards compatibility, when namespace isn't specified in dataSourceRef, both fields (dataSource and dataSourceRef) will be set to the same value automatically if one of them is empty and the other is non-empty. When namespace is specified in dataSourceRef, dataSource isn't set to the same value and must be empty. There are three important differences between dataSource and dataSourceRef: * While dataSource only allows two specific types of objects, dataSourceRef
  allows any non-core object, as well as PersistentVolumeClaim objects.
* While dataSource ignores disallowed values (dropping them), dataSourceRef
  preserves all values, and generates an error if a disallowed value is
  specified.
* While dataSource only allows local objects, dataSourceRef allows objects
  in any namespaces.
(Beta) Using this field requires the AnyVolumeDataSource feature gate to be enabled. (Alpha) Using the namespace field of dataSourceRef requires the CrossNamespaceVolumeDataSource feature gate to be enabled.

#### resources

resources represents the minimum resources the volume should have. Users are allowed to specify resource requirements that are lower than previous value but must still be higher than capacity recorded in the status field of the claim. More info: https://kubernetes.io/docs/concepts/storage/persistent-volumes#resources

#### selector

selector is a label query over volumes to consider for binding.

#### storageClassName

_Type:_ `string`

storageClassName is the name of the StorageClass required by the claim. More info: https://kubernetes.io/docs/concepts/storage/persistent-volumes#class-1

#### volumeAttributesClassName

_Type:_ `string`

volumeAttributesClassName may be used to set the VolumeAttributesClass used by this claim. If specified, the CSI driver will create or update the volume with the attributes defined in the corresponding VolumeAttributesClass. This has a different purpose than storageClassName, it can be changed after the claim is created. An empty string or nil value indicates that no VolumeAttributesClass will be applied to the claim. If the claim enters an Infeasible error state, this field can be reset to its previous value (including nil) to cancel the modification. If the resource referred to by volumeAttributesClass does not exist, this PersistentVolumeClaim will be set to a Pending state, as reflected by the modifyVolumeStatus field, until such as a resource exists. More info: https://kubernetes.io/docs/concepts/storage/volume-attributes-classes/

#### volumeMode

_Type:_ `string`

volumeMode defines what type of volume is required by the claim. Value of Filesystem is implied when not included in claim spec.

#### volumeName

_Type:_ `string`

volumeName is the binding reference to the PersistentVolume backing this claim.

### k8s.io.api.core.v1.PersistentVolumeClaimTemplate

Embedded PVC template used inside `EphemeralVolumeSource` to
materialize a per-Pod PVC bound to the Pod's lifecycle. Two fields:
`metadata` (labels and annotations applied to the generated PVC —
`name` is server-set as `<pod-name>-<volume-name>` and any name in
the template is rejected) and `spec` (a PersistentVolumeClaimSpec
describing the storage to provision). The kubelet creates the PVC
when the Pod is admitted and deletes it when the Pod is deleted —
it does NOT survive Pod rescheduling or Deployment rollouts.

**When to use:** Use only inside `EphemeralVolumeSource.volumeClaimTemplate` — that
is its sole valid home. For shared durable storage, use a top-level
PVC and reference it via PersistentVolumeClaimVolumeSource. For

`volumeClaimTemplates` (different API shape, similar concept).

**Related fields:**
- `volumeClaimTemplate`
- `persistentVolumeClaim`
- `persistentVolumeClaims`

**Common mistakes:**
- Setting `metadata.name` in the template — Kubernetes rejects the Pod; the name is server-generated.
- Forgetting that the generated PVC is deleted with the Pod — Deployment rollouts kill old Pods which destroys their ephemeral PVCs, so data does not survive rollouts.
- Requesting `ReadWriteMany` in a single-pod ephemeral template — supported but pointless; only one Pod ever attaches an ephemeral PVC, so RWO is correct.

**Example:**

```yaml
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
  storageClassName: standard
```

#### metadata

May contain labels and annotations that will be copied into the PVC when creating it. No other fields are allowed and will be rejected during validation.

#### spec

The specification for the PersistentVolumeClaim. The entire content is copied unchanged into the PVC that gets created from this template. The same fields as in a PersistentVolumeClaim are also valid here.

### k8s.io.api.core.v1.PersistentVolumeClaimVolumeSource

Mounts an existing PersistentVolumeClaim by name. `claimName`
references a PVC in the same namespace as the Pod; the PVC's bound
PersistentVolume is what actually mounts. `readOnly: true` forces a
read-only mount even if the PVC's access mode would allow writes
(rare — typically controlled via `volumeMounts[].readOnly`). The
PVC's `accessModes` determine whether multiple Pods can attach
concurrently — `ReadWriteOnce` means one node at a time (multiple
Pods on the same node may share), `ReadWriteMany` means multiple
nodes, `ReadOnlyMany` means multiple read-only mounts. Some
providers also support `ReadWriteOncePod` (Kubernetes 1.27+) for
strict single-Pod attachment.

**When to use:** Use for durable storage that must survive Pod restarts or
rescheduling. The chart's `persistentVolumeClaims:` top-level field
lets you declare PVCs alongside the Deployment; this volume source
then references them by name. For per-Pod ephemeral PVCs (created
and destroyed with the Pod), use an `EphemeralVolumeSource` instead.

**Related fields:**
- `persistentVolumeClaims`
- `volumes`
- `volumeMounts`

**Common mistakes:**
- Pointing `claimName` at a PVC in another namespace — silently fails; PVCs are namespace-local.
- Attaching a `ReadWriteOnce` PVC to a multi-replica Deployment without per-replica affinity — every replica fights for the same volume, only one starts, the rest are Pending.
- Setting `readOnly: true` here while the PVC's `accessModes` doesn't include `ReadOnlyMany` and the workload expects write access — the mount succeeds read-only and writes fail at runtime with EROFS.

**Example:**

```yaml
claimName: app-data
```

#### claimName

_Type:_ `string`

claimName is the name of a PersistentVolumeClaim in the same namespace as the pod using this volume. More info: https://kubernetes.io/docs/concepts/storage/persistent-volumes#persistentvolumeclaims

#### readOnly

_Type:_ `boolean`

readOnly Will force the ReadOnly setting in VolumeMounts. Default false.

### k8s.io.api.core.v1.PodAffinity

Co-location rules — schedule this Pod *near* other Pods matching a
label selector inside the same topology domain. Two flavors:
`requiredDuringSchedulingIgnoredDuringExecution` (hard — Pending if
no matching domain exists) and
`preferredDuringSchedulingIgnoredDuringExecution` (soft, weighted
preference). PodAffinity is expensive at scale: the scheduler
enumerates every Pod in the cluster (or in `namespaces`) to evaluate
the selector, which becomes a bottleneck above a few thousand pods.
Use `topologySpreadConstraints` for general spreading and reserve
PodAffinity for cases that genuinely need co-location (cache locality
with a specific peer service).

**When to use:** Use only when co-location is a real performance win — e.g. an app
Pod that benefits from same-host caching alongside a cache Pod.
For "spread my pods evenly" prefer `topologySpreadConstraints`;
for "don't put two of my pods on the same host" use `podAntiAffinity`.

**Related fields:**
- `podAntiAffinity`
- `topologySpreadConstraints`
- `affinity`

**Common mistakes:**
- Forgetting `topologyKey` — it has no default and the scheduler rejects the Pod at admission.
- Using `required...` PodAffinity in a multi-zone cluster while the target service runs in a single zone — pods become Pending in other zones with no recovery.
- Setting PodAffinity at large scale and seeing scheduler latency spike — the selector evaluates against every Pod in scope; constrain via `namespaces` or `namespaceSelector` or move to topologySpread.

**Example:**

```yaml
preferredDuringSchedulingIgnoredDuringExecution:
- podAffinityTerm:
    labelSelector:
      matchLabels:
        app.kubernetes.io/name: redis
    topologyKey: kubernetes.io/hostname
  weight: 50
```

#### preferredDuringSchedulingIgnoredDuringExecution

_Type:_ `array`

The scheduler will prefer to schedule pods to nodes that satisfy the affinity expressions specified by this field, but it may choose a node that violates one or more of the expressions. The node that is most preferred is the one with the greatest sum of weights, i.e. for each node that meets all of the scheduling requirements (resource request, requiredDuringScheduling affinity expressions, etc.), compute a sum by iterating through the elements of this field and adding "weight" to the sum if the node has pods which matches the corresponding podAffinityTerm; the node(s) with the highest sum are the most preferred.

#### requiredDuringSchedulingIgnoredDuringExecution

_Type:_ `array`

If the affinity requirements specified by this field are not met at scheduling time, the pod will not be scheduled onto the node. If the affinity requirements specified by this field cease to be met at some point during pod execution (e.g. due to a pod label update), the system may or may not try to eventually evict the pod from its node. When there are multiple elements, the lists of nodes corresponding to each podAffinityTerm are intersected, i.e. all terms must be satisfied.

### k8s.io.api.core.v1.PodAffinityTerm

A single pod-affinity / pod-anti-affinity rule. `labelSelector` picks
target Pods; `topologyKey` is the node label that defines the
topology domain (`kubernetes.io/hostname` for "same node",
`topology.kubernetes.io/zone` for "same zone"); `namespaces` /
`namespaceSelector` scope which namespaces the selector evaluates
against (default: this Pod's namespace only). `matchLabelKeys` /
`mismatchLabelKeys` (Kubernetes 1.29+) compute per-Pod label values
and AND them into the selector — useful for "co-locate with Pods of
the same revision" without templating the label values.

**When to use:** Use inside PodAffinity / PodAntiAffinity entries. For "spread N copies
across zones evenly" the modern alternative is
`topologySpreadConstraints` — easier to reason about than weighted
anti-affinity. Explicitly set `namespaces` or `namespaceSelector`
when you intentionally want cross-namespace matching.

**Related fields:**
- `podAffinity`
- `podAntiAffinity`
- `topologyKey`
- `topologySpreadConstraints`

**Common mistakes:**
- Omitting `topologyKey` — the scheduler rejects the Pod at admission with `topologyKey is required`.
- Using `topologyKey: kubernetes.io/hostname` with `requiredDuringScheduling...` and `replicas` > number of nodes — the surplus replicas become Pending forever.
- Forgetting that the default namespace scope is *this Pod's namespace only* — selectors against a Service in another namespace silently match nothing.

**Example:**

```yaml
labelSelector:
  matchLabels:
    app.kubernetes.io/name: my-app
topologyKey: kubernetes.io/hostname
```

#### labelSelector

A label query over a set of resources, in this case pods. If it's null, this PodAffinityTerm matches with no Pods.

#### matchLabelKeys

_Type:_ `array`

MatchLabelKeys is a set of pod label keys to select which pods will be taken into consideration. The keys are used to lookup values from the incoming pod labels, those key-value labels are merged with `labelSelector` as `key in (value)` to select the group of existing pods which pods will be taken into consideration for the incoming pod's pod (anti) affinity. Keys that don't exist in the incoming pod labels will be ignored. The default value is empty. The same key is forbidden to exist in both matchLabelKeys and labelSelector. Also, matchLabelKeys cannot be set when labelSelector isn't set.

#### mismatchLabelKeys

_Type:_ `array`

MismatchLabelKeys is a set of pod label keys to select which pods will be taken into consideration. The keys are used to lookup values from the incoming pod labels, those key-value labels are merged with `labelSelector` as `key notin (value)` to select the group of existing pods which pods will be taken into consideration for the incoming pod's pod (anti) affinity. Keys that don't exist in the incoming pod labels will be ignored. The default value is empty. The same key is forbidden to exist in both mismatchLabelKeys and labelSelector. Also, mismatchLabelKeys cannot be set when labelSelector isn't set.

#### namespaceSelector

A label query over the set of namespaces that the term applies to. The term is applied to the union of the namespaces selected by this field and the ones listed in the namespaces field. null selector and null or empty namespaces list means "this pod's namespace". An empty selector ({}) matches all namespaces.

#### namespaces

_Type:_ `array`

namespaces specifies a static list of namespace names that the term applies to. The term is applied to the union of the namespaces listed in this field and the ones selected by namespaceSelector. null or empty namespaces list and null namespaceSelector means "this pod's namespace".

#### topologyKey

_Type:_ `string`

This pod should be co-located (affinity) or not co-located (anti-affinity) with the pods matching the labelSelector in the specified namespaces, where co-located is defined as running on a node whose value of the label with key topologyKey matches that of any node on which any of the selected pods is running. Empty topologyKey is not allowed.

### k8s.io.api.core.v1.PodAntiAffinity

Anti-co-location rules — keep this Pod *away from* other Pods matching
a label selector within the same topology domain. Two flavors,
`requiredDuringSchedulingIgnoredDuringExecution` (hard — Pending if
no domain free of conflicting Pods exists) and
`preferredDuringSchedulingIgnoredDuringExecution` (weighted soft
preference). The chart's `autoCreateSoftAntiAffinity: true` injects
a soft `kubernetes.io/hostname` anti-affinity term automatically;
anything you set here is appended rather than overriding the chart's
injection. Like PodAffinity, evaluation is expensive at scale —
`topologySpreadConstraints` is the modern, more predictable
alternative for general spread.

**When to use:** Use `required...` only when colocation is *unsafe* (singleton
enforcement, license capping). For general HA spreading
`topologySpreadConstraints` is clearer and the scheduler is faster.
The chart's `autoCreateSoftAntiAffinity` covers the common
per-host soft spread without writing this block.

**Related fields:**
- `podAffinity`
- `topologySpreadConstraints`
- `autoCreateSoftAntiAffinity`

**Common mistakes:**
- Required hostname anti-affinity with `replicas` > number of nodes — surplus replicas stay Pending; switch to preferred or use topologySpread.
- Setting hostname anti-affinity while the workload is itself a DaemonSet candidate — DaemonSet semantics already enforce one per node and the affinity is redundant.
- Combining this with the chart's `autoCreateSoftAntiAffinity` and being surprised at duplicated rules — both are applied; remove `autoCreateSoftAntiAffinity` if you're writing the affinity by hand.

**Example:**

```yaml
preferredDuringSchedulingIgnoredDuringExecution:
- podAffinityTerm:
    labelSelector:
      matchLabels:
        app.kubernetes.io/name: my-app
    topologyKey: kubernetes.io/hostname
  weight: 100
```

#### preferredDuringSchedulingIgnoredDuringExecution

_Type:_ `array`

The scheduler will prefer to schedule pods to nodes that satisfy the anti-affinity expressions specified by this field, but it may choose a node that violates one or more of the expressions. The node that is most preferred is the one with the greatest sum of weights, i.e. for each node that meets all of the scheduling requirements (resource request, requiredDuringScheduling anti-affinity expressions, etc.), compute a sum by iterating through the elements of this field and subtracting "weight" from the sum if the node has pods which matches the corresponding podAffinityTerm; the node(s) with the highest sum are the most preferred.

#### requiredDuringSchedulingIgnoredDuringExecution

_Type:_ `array`

If the anti-affinity requirements specified by this field are not met at scheduling time, the pod will not be scheduled onto the node. If the anti-affinity requirements specified by this field cease to be met at some point during pod execution (e.g. due to a pod label update), the system may or may not try to eventually evict the pod from its node. When there are multiple elements, the lists of nodes corresponding to each podAffinityTerm are intersected, i.e. all terms must be satisfied.

### k8s.io.api.core.v1.PodCertificateProjection

Projected volume entry that obtains a short-lived X.509 certificate
bound to the Pod identity and writes it to a path.

**Example:**

```yaml
{}
```

#### certificateChainPath

_Type:_ `string`

Write the certificate chain at this path in the projected volume.

Most applications should use credentialBundlePath.  When using keyPath and certificateChainPath, your application needs to check that the key and leaf certificate are consistent, because it is possible to read the files mid-rotation.

#### credentialBundlePath

_Type:_ `string`

Write the credential bundle at this path in the projected volume.

The credential bundle is a single file that contains multiple PEM blocks. The first PEM block is a PRIVATE KEY block, containing a PKCS#8 private key.

The remaining blocks are CERTIFICATE blocks, containing the issued certificate chain from the signer (leaf and any intermediates).

Using credentialBundlePath lets your Pod's application code make a single atomic read that retrieves a consistent key and certificate chain.  If you project them to separate files, your application code will need to additionally check that the leaf certificate was issued to the key.

#### keyPath

_Type:_ `string`

Write the key at this path in the projected volume.

Most applications should use credentialBundlePath.  When using keyPath and certificateChainPath, your application needs to check that the key and leaf certificate are consistent, because it is possible to read the files mid-rotation.

#### keyType

_Type:_ `string`

The type of keypair Kubelet will generate for the pod.

Valid values are "RSA3072", "RSA4096", "ECDSAP256", "ECDSAP384", "ECDSAP521", and "ED25519".

#### maxExpirationSeconds

_Type:_ `integer`

maxExpirationSeconds is the maximum lifetime permitted for the certificate.

Kubelet copies this value verbatim into the PodCertificateRequests it generates for this projection.

If omitted, kube-apiserver will set it to 86400(24 hours). kube-apiserver will reject values shorter than 3600 (1 hour).  The maximum allowable value is 7862400 (91 days).

The signer implementation is then free to issue a certificate with any lifetime *shorter* than MaxExpirationSeconds, but no shorter than 3600 seconds (1 hour).  This constraint is enforced by kube-apiserver. `kubernetes.io` signers will never issue certificates with a lifetime longer than 24 hours.

#### signerName

_Type:_ `string`

Kubelet's generated CSRs will be addressed to this signer.

#### userAnnotations

_Type:_ `object`

userAnnotations allow pod authors to pass additional information to the signer implementation.  Kubernetes does not restrict or validate this metadata in any way.

These values are copied verbatim into the `spec.unverifiedUserAnnotations` field of the PodCertificateRequest objects that Kubelet creates.

Entries are subject to the same validation as object metadata annotations, with the addition that all keys must be domain-prefixed. No restrictions are placed on values, except an overall size limitation on the entire field.

Signers should document the keys and values they support. Signers should deny requests that contain keys they do not recognize.

### k8s.io.api.core.v1.PodDNSConfig

Custom resolver configuration applied to the Pod's `/etc/resolv.conf`,
composed of `nameservers`, `searches`, and `options` lists. The exact
merge behavior is governed by the Pod's `dnsPolicy`: with
`ClusterFirst` (the default) the cluster nameservers come first and
this block appends; with `dnsPolicy: None` this block is the entire
configuration and at least one nameserver is required. The chart does
not currently auto-set `dnsPolicy: None`, so setting `dnsConfig`
alone produces an additive override rather than a full replacement.
Kubernetes enforces hard limits — a maximum of 3 nameservers and 6
search domains — and rejects the Pod at admission if exceeded.

**When to use:** Use to tighten `ndots`, shorten `timeout`, or add a corporate search
domain. For complete control over resolution (e.g. bypassing cluster
DNS entirely) pair this with `dnsPolicy: None`. If you only need to
keep the host's resolver, prefer `dnsPolicy: Default` over carrying a
handcrafted dnsConfig.

**Related fields:**
- `dnsPolicy`
- `hostNetwork`
- `hostAliases`

**Common mistakes:**
- Listing more than 3 `nameservers` (or more than 6 `searches`) — Kubernetes rejects the Pod at admission time with a validation error, the workload never starts.
- Writing `search:` (singular) instead of `searches:` — the field is silently ignored, no extra search domains are applied, and the misconfiguration is invisible until you `kubectl exec` and inspect `/etc/resolv.conf`.
- Setting `dnsConfig` while leaving `dnsPolicy: ClusterFirst` and expecting cluster DNS to be replaced — it isn't, the entries are merely appended; full replacement requires `dnsPolicy: None`.

**Example:**

```yaml
nameservers:
- 1.1.1.1
- 8.8.8.8
options:
- name: ndots
  value: '2'
- name: timeout
  value: '1'
```

#### nameservers

_Type:_ `array`

A list of DNS name server IP addresses. This will be appended to the base nameservers generated from DNSPolicy. Duplicated nameservers will be removed.

#### options

_Type:_ `array`

A list of DNS resolver options. This will be merged with the base options generated from DNSPolicy. Duplicated entries will be removed. Resolution options given in Options will override those that appear in the base DNSPolicy.

#### searches

_Type:_ `array`

A list of DNS search domains for host-name lookup. This will be appended to the base search paths generated from DNSPolicy. Duplicated search paths will be removed.

### k8s.io.api.core.v1.PodDNSConfigOption

One `/etc/resolv.conf` option entry. `name` is the option name
(`ndots`, `timeout`, `attempts`, `single-request-reopen`,
`rotate`, etc.); `value` is the option's value as a string —
options like `single-request-reopen` that take no value should
omit `value`. Multiple entries combine into resolv.conf's
`options ndots:2 timeout:1` line. Used inside
`PodDNSConfig.options`.

**When to use:** The most common tuning is lowering `ndots` from the cluster default
of 5 to 2 — at 5, every short hostname triggers multiple cluster-
DNS lookups via the search domain expansion, which is a measurable
latency source on chatty workloads. Set `timeout: "1"` to fail
faster on slow DNS.

**Related fields:**
- `dnsConfig`
- `options`
- `dnsPolicy`

**Common mistakes:**
- Omitting the quotes on numeric values (`value: 2`) — YAML parses it as an integer; resolv.conf expects strings and most kubelets reject the int variant.
- Setting `ndots: "1"` while heavily using short cluster-DNS names — `ndots:1` skips search-domain expansion for any name with a dot, breaking common short references like `service.namespace`.
- Setting options that don't take a value (`single-request-reopen`) with `value: ""` — works on some Kubernetes versions, is rejected on others; safest to omit `value` entirely.

**Example:**

```yaml
name: ndots
value: '2'
```

#### name

_Type:_ `string`

Name is this DNS resolver option's name. Required.

#### value

_Type:_ `string`

Value is this DNS resolver option's value.

### k8s.io.api.core.v1.PodSecurityContext

Pod-level security settings that apply to every container in the Pod
unless a container's own `securityContext` overrides them. Covers
`runAsUser`, `runAsGroup`, `runAsNonRoot`, `fsGroup`,
`supplementalGroups`, `sysctls`, `seLinuxOptions`, the `seccompProfile`,
and Linux/Windows-specific options. Note that this chart does NOT
default `runAsNonRoot: true` — if your image is built to run as a
non-root user you must set it explicitly here (or at the container
level) to keep the kubelet from starting a process as UID 0. The
`fsGroup` setting recursively chowns the contents of every volume
mounted into the Pod at startup, which can be very expensive on large
PersistentVolumes; pair with `fsGroupChangePolicy: OnRootMismatch` to
avoid the per-pod-start scan.

**When to use:** Use this Pod-level block for settings that apply to all containers
(UID/GID, volume `fsGroup`, sysctls). For container-specific knobs
like `readOnlyRootFilesystem`, `capabilities`, or `privileged`, use
the per-container `securityContext` instead — those fields do not
exist at the Pod level.

**Related fields:**
- `securityContext`
- `containers[].securityContext`
- `volumes`

**Common mistakes:**
- Setting `fsGroup` on a Pod whose container also has `readOnlyRootFilesystem: true` while expecting the chown to succeed on the root FS — the chown only touches mounted volumes, but operators routinely confuse the two and wonder why write fails.
- Putting `readOnlyRootFilesystem` or `capabilities` here — those are container-level only; placed at the Pod level they are silently dropped (or rejected, depending on the validator).
- Setting `runAsUser: 0` (root) together with a `RuntimeDefault` seccomp profile and hoping the profile contains root — `runAsNonRoot: true` would have caught the slip; without it the Pod runs as root and gets through.

**Example:**

```yaml
fsGroup: 2000
runAsGroup: 1000
runAsNonRoot: true
runAsUser: 1000
seccompProfile:
  type: RuntimeDefault
```

#### appArmorProfile

appArmorProfile is the AppArmor options to use by the containers in this pod. Note that this field cannot be set when spec.os.name is windows.

#### fsGroup

_Type:_ `integer`

A special supplemental group that applies to all containers in a pod. Some volume types allow the Kubelet to change the ownership of that volume to be owned by the pod:

1. The owning GID will be the FSGroup 2. The setgid bit is set (new files created in the volume will be owned by FSGroup) 3. The permission bits are OR'd with rw-rw----

If unset, the Kubelet will not modify the ownership and permissions of any volume. Note that this field cannot be set when spec.os.name is windows.

#### fsGroupChangePolicy

_Type:_ `string`

fsGroupChangePolicy defines behavior of changing ownership and permission of the volume before being exposed inside Pod. This field will only apply to volume types which support fsGroup based ownership(and permissions). It will have no effect on ephemeral volume types such as: secret, configmaps and emptydir. Valid values are "OnRootMismatch" and "Always". If not specified, "Always" is used. Note that this field cannot be set when spec.os.name is windows.

#### runAsGroup

_Type:_ `integer`

The GID to run the entrypoint of the container process. Uses runtime default if unset. May also be set in SecurityContext.  If set in both SecurityContext and PodSecurityContext, the value specified in SecurityContext takes precedence for that container. Note that this field cannot be set when spec.os.name is windows.

#### runAsNonRoot

_Type:_ `boolean`

Indicates that the container must run as a non-root user. If true, the Kubelet will validate the image at runtime to ensure that it does not run as UID 0 (root) and fail to start the container if it does. If unset or false, no such validation will be performed. May also be set in SecurityContext.  If set in both SecurityContext and PodSecurityContext, the value specified in SecurityContext takes precedence.

#### runAsUser

_Type:_ `integer`

The UID to run the entrypoint of the container process. Defaults to user specified in image metadata if unspecified. May also be set in SecurityContext.  If set in both SecurityContext and PodSecurityContext, the value specified in SecurityContext takes precedence for that container. Note that this field cannot be set when spec.os.name is windows.

#### seLinuxChangePolicy

_Type:_ `string`

seLinuxChangePolicy defines how the container's SELinux label is applied to all volumes used by the Pod. It has no effect on nodes that do not support SELinux or to volumes does not support SELinux. Valid values are "MountOption" and "Recursive".

"Recursive" means relabeling of all files on all Pod volumes by the container runtime. This may be slow for large volumes, but allows mixing privileged and unprivileged Pods sharing the same volume on the same node.

"MountOption" mounts all eligible Pod volumes with `-o context` mount option. This requires all Pods that share the same volume to use the same SELinux label. It is not possible to share the same volume among privileged and unprivileged Pods. Eligible volumes are in-tree FibreChannel and iSCSI volumes, and all CSI volumes whose CSI driver announces SELinux support by setting spec.seLinuxMount: true in their CSIDriver instance. Other volumes are always re-labelled recursively. "MountOption" value is allowed only when SELinuxMount feature gate is enabled.

### k8s.io.api.core.v1.PreferredSchedulingTerm

Weighted soft node-affinity entry — pairs a `weight` (1–100) with a
`preference` (one NodeSelectorTerm). The scheduler ranks candidate
nodes by summing weights of every matching preferred term, and picks
the highest-scoring node. Used inside
`nodeAffinity.preferredDuringSchedulingIgnoredDuringExecution[]`.
Weight is *additive across terms*: two non-overlapping preferred
terms with weights 30 and 50 contribute 80 to nodes matching both.

**When to use:** Use to express "I'd like X if available, but it's not a hard
requirement." Pair multiple terms with different weights to encode
tiers of preference (e.g. zone-local > region-local > anywhere).
For a hard rule, use NodeSelector inside `requiredDuringScheduling...`.

**Related fields:**
- `nodeAffinity`
- `preferredDuringSchedulingIgnoredDuringExecution`
- `preference`

**Common mistakes:**
- Setting `weight: 0` — silently dropped; the scheduler treats it as no preference. Weight must be 1–100.
- Inflating weights to dominate other affinity considerations — weights from PodAffinity and NodeAffinity sum together, but a single weight > 100 is rejected by the API.
- Forgetting that preferred terms are *soft* — the scheduler will place the Pod elsewhere if all preferred nodes are full or tainted, which can be surprising.

**Example:**

```yaml
preference:
  matchExpressions:
  - key: instance-type
    operator: In
    values:
    - c5.large
    - c5.xlarge
weight: 80
```

#### preference

A node selector term, associated with the corresponding weight.

#### weight

_Type:_ `integer`

Weight associated with matching the corresponding nodeSelectorTerm, in the range 1-100.

### k8s.io.api.core.v1.ProjectedVolumeSource

Combines several source types into a single mount. `sources[]` lists
VolumeProjection entries — each one a `configMap`, `secret`,
`downwardAPI`, `serviceAccountToken`, `clusterTrustBundle`, or
`podCertificate` projection. All entries are laid out under the
single mount path, with each projection writing its files relative
to that path. `defaultMode` applies to files without per-item
modes. The big use case: bound (audience-scoped, expiring) service
account tokens via `serviceAccountToken` projections — the kubelet
refreshes them automatically. The legacy `secretName-only`
ServiceAccount token mount is being phased out in favor of this
shape.

**When to use:** Use when you need to expose configuration *and* secrets *and*
identity at the same mount path (e.g. the standard layout expected
by SPIRE/SPIFFE workloads, or a sidecar that reads multiple
categories of data from one directory). Avoid for simple
single-source cases; the dedicated `configMap`/`secret` volume
sources are simpler.

**Related fields:**
- `sources`
- `volumes`
- `serviceAccountToken`

**Common mistakes:**
- Two projections writing files to the same target path inside the volume — the second silently overwrites the first; ensure `path:` values are unique within `sources`.
- Forgetting that the audience-bound token expires (`expirationSeconds` default ~1h) — the application must reload the token from disk; the kubelet rewrites the file but processes that read it once at startup won't pick up the refresh.
- Putting only one source in `sources[]` — works, but adds boilerplate; use the dedicated volume source (ConfigMapVolumeSource, SecretVolumeSource) for single-source cases.

**Example:**

```yaml

    path: vault-token

      path: config/app.yaml
    name: app-config
```

_Type:_ `integer`

### k8s.io.api.core.v1.SeccompProfile

Seccomp (secure computing) filter applied to the container's syscalls.
`type` is the discriminator: `RuntimeDefault` uses the container
runtime's curated default profile (the recommended baseline);
`Unconfined` disables seccomp entirely (only safe for trusted
workloads); `Localhost` uses a profile file at
`<seccompProfileRoot>/<localhostProfile>` on the node (path must be
pre-deployed by the node operator). Pod Security Standard
`restricted` requires `RuntimeDefault` or `Localhost`; `Unconfined`
is allowed only by `privileged`. Can be set at Pod level (applies
to every container) or per-container.

**When to use:** Use `RuntimeDefault` for nearly every workload — it blocks the
historically-exploited syscalls without breaking common apps. Use
`Localhost` only when you've authored a custom profile (audit a
run, capture syscalls, generate). Reserve `Unconfined` for
genuinely privileged workloads (debugging tools, runtimes-in-runtime).

**Related fields:**
- `securityContext`
- `seccompProfile`
- `allowPrivilegeEscalation`

**Common mistakes:**
- Setting `type: Localhost` without `localhostProfile` (or pointing at a path the node doesn't have) — Pod fails to start with `CreateContainerError`.
- Setting `Unconfined` to work around a permission denial without identifying the missing syscall — better to use `audit2allow`-style tooling against a `Localhost` profile.
- Setting seccomp at the Pod level and a stricter profile on one container, expecting them to compose — the container-level wins; the Pod-level setting is replaced entirely for that container.

**Example:**

```yaml
type: RuntimeDefault
```

#### localhostProfile

_Type:_ `string`

localhostProfile indicates a profile defined in a file on the node should be used. The profile must be preconfigured on the node to work. Must be a descending path, relative to the kubelet's configured seccomp profile location. Must be set if type is "Localhost". Must NOT be set for any other type.

#### type

_Type:_ `string`

type indicates which kind of seccomp profile will be applied. Valid options are:

Localhost - a profile defined in a file on the node should be used. RuntimeDefault - the container runtime default profile should be used. Unconfined - no profile should be applied.

### k8s.io.api.core.v1.SecretVolumeSource

### k8s.io.api.core.v1.SecurityContext

Container-level security settings. Covers `runAsUser`, `runAsGroup`,
`runAsNonRoot`, `capabilities` (add/drop), `privileged`,
`readOnlyRootFilesystem`, `allowPrivilegeEscalation`, `seccompProfile`,
`appArmorProfile`, `seLinuxOptions`, and `procMount`. For fields that
also exist at the Pod level (`runAsUser`, `runAsGroup`, `runAsNonRoot`,
seccomp/AppArmor/SELinux profiles), the container-level setting wins.
Fields that only exist here (`capabilities`, `privileged`,
`readOnlyRootFilesystem`, `allowPrivilegeEscalation`) cannot be set
at the Pod level. The chart wires this through unchanged — set it on
each container that needs hardening.

**When to use:** Use here for container-only knobs (`capabilities`, `privileged`,
`readOnlyRootFilesystem`, `allowPrivilegeEscalation`) and for any
shared field where you want to override the Pod-level default for
one container. For settings that should cover every container in the
Pod (UID/GID, volume fsGroup, sysctls), prefer the Pod-level
`securityContext` to avoid duplication.

**Related fields:**
- `securityContext`
- `containers`
- `volumes`

**Common mistakes:**
- Setting `readOnlyRootFilesystem: true` without mounting an `emptyDir` at `/tmp` (and other write paths your app uses) — the container starts but the first write fails with EROFS, often inside a library you don't control.
- Using `capabilities: { add: [...] }` without an explicit `drop: [ALL]` — you keep the full default capability set and merely add to it, which defeats the purpose of granting a single capability.
- Granting `privileged: true` to work around a missing capability — almost always wrong; the right answer is `capabilities.add: [SPECIFIC_CAP]` so the rest of the kernel surface stays restricted.

**Example:**

```yaml
allowPrivilegeEscalation: false
capabilities:
  drop:
  - ALL
readOnlyRootFilesystem: true
runAsNonRoot: true
```

#### allowPrivilegeEscalation

_Type:_ `boolean`

AllowPrivilegeEscalation controls whether a process can gain more privileges than its parent process. This bool directly controls if the no_new_privs flag will be set on the container process. AllowPrivilegeEscalation is true always when the container is: 1) run as Privileged 2) has CAP_SYS_ADMIN Note that this field cannot be set when spec.os.name is windows.

#### appArmorProfile

appArmorProfile is the AppArmor options to use by this container. If set, this profile overrides the pod's appArmorProfile. Note that this field cannot be set when spec.os.name is windows.

#### capabilities

The capabilities to add/drop when running containers. Defaults to the default set of capabilities granted by the container runtime. Note that this field cannot be set when spec.os.name is windows.

#### privileged

_Type:_ `boolean`

Run container in privileged mode. Processes in privileged containers are essentially equivalent to root on the host. Defaults to false. Note that this field cannot be set when spec.os.name is windows.

#### procMount

_Type:_ `string`

procMount denotes the type of proc mount to use for the containers. The default value is Default which uses the container runtime defaults for readonly paths and masked paths. This requires the ProcMountType feature flag to be enabled. Note that this field cannot be set when spec.os.name is windows.

#### readOnlyRootFilesystem

_Type:_ `boolean`

Whether this container has a read-only root filesystem. Default is false. Note that this field cannot be set when spec.os.name is windows.

#### runAsGroup

_Type:_ `integer`

The GID to run the entrypoint of the container process. Uses runtime default if unset. May also be set in PodSecurityContext.  If set in both SecurityContext and PodSecurityContext, the value specified in SecurityContext takes precedence. Note that this field cannot be set when spec.os.name is windows.

#### runAsNonRoot

_Type:_ `boolean`

Indicates that the container must run as a non-root user. If true, the Kubelet will validate the image at runtime to ensure that it does not run as UID 0 (root) and fail to start the container if it does. ### k8s.io.api.core.v1.ServiceAccountTokenProjection

Requests a *bound* (audience-scoped, time-bound, automatically
rotated) ServiceAccount token and writes it to a file. `audience` is
the intended recipient (e.g. `vault`, `https://kubernetes.default.svc`);
`expirationSeconds` is the requested lifetime (default 1h, minimum
10 minutes — values below are clamped); `path` is the relative file
inside the projected volume. The kubelet rotates the token in place
well before expiry, so consumers must re-read the file periodically.
This is the modern replacement for the legacy automount-token Secret
pattern — bound tokens are audience-validated, can't be reused
against unintended services, and don't sit in etcd long-term.

**When to use:** Use whenever a workload authenticates to an external service that
verifies the token's audience (Vault, AWS IAM-OIDC, custom OIDC
verifiers). Audience must match what the verifier expects;
omitting it issues a default-audience token that most external
verifiers will reject.

**Related fields:**
- `sources`
- `projected`
- `serviceAccountToken`

**Common mistakes:**
- Caching the token in application memory and never re-reading from disk — the kubelet rotates the file before expiry, but the in-memory copy stops being valid silently.
- Setting `expirationSeconds` below 600 — the API server enforces a 10-minute minimum and rounds up, the configured value is misleading.
- Forgetting that `audience` is required — without it the token is issued with the API server's default audience, which external verifiers reject.

**Example:**

```yaml
audience: vault
expirationSeconds: 3600
path: vault-token
```

#### audience

_Type:_ `string`

audience is the intended audience of the token. A recipient of a token must identify itself with an identifier specified in the audience of the token, and otherwise should reject the token. The audience defaults to the identifier of the apiserver.

#### expirationSeconds

_Type:_ `integer`

expirationSeconds is the requested duration of validity of the service account token. As the token approaches expiration, the kubelet volume plugin will proactively rotate the service account token. The kubelet will start trying to rotate the token if the token is older than 80 percent of its time to live or if the token is older than 24 hours.Defaults to 1 hour and must be at least 10 minutes.

#### path

_Type:_ `string`

path is the path relative to the mount point of the file to project the token into.

### k8s.io.api.core.v1.SleepAction

LifecycleHandler action that pauses for `seconds` seconds. Added in
Kubernetes 1.29 (beta in 1.30, GA in 1.32) as a cleaner expression
of the long-standing "preStop: exec sleep N" pattern — no shell, no
process spawn, no path to a sleep binary. The pause is implemented
kubelet-side as a timer, not a sleep process. Only valid inside
`lifecycle.preStop` or `lifecycle.postStart`; not a probe action.

**When to use:** Use as the canonical preStop hook to give the kube-proxy /
service-mesh / ingress controller time to remove this Pod from
endpoint lists before the container receives SIGTERM. 10–30 seconds
is typical; tune to your endpoint-removal propagation. Ensure
`terminationGracePeriodSeconds` is at least `seconds + app shutdown
time` or SIGKILL preempts the work.

**Related fields:**
- `lifecycle`
- `preStop`
- `terminationGracePeriodSeconds`

**Common mistakes:**
- Sleeping longer than `terminationGracePeriodSeconds` — the kubelet SIGKILLs at expiry, the sleep is interrupted, in-flight work is lost. Always: sleep < grace period.
- Using `SleepAction` on Kubernetes < 1.29 — the field doesn't exist in older clusters; fall back to `exec: { command: [sleep, "15"] }`.
- Adding both a `sleep` preStop and a separate graceful-shutdown signal in the app — the app receives SIGTERM only *after* the sleep completes; double-sleep accidentally adds wait time.

**Example:**

```yaml
seconds: 15
```

#### seconds

_Type:_ `integer`

Seconds is the number of seconds to sleep.

### k8s.io.api.core.v1.Sysctl

One sysctl `{name, value}` pair applied to the Pod by the kubelet at
container start. Only valid inside `PodSecurityContext.sysctls`.
Kubernetes splits sysctls into two categories: *safe* (namespaced,
cannot affect other Pods or the host — e.g. `net.ipv4.ip_local_port_range`,
`kernel.shm_rmid_forced`, a small whitelist) which work everywhere,
and *unsafe* (everything else — `net.core.somaxconn`,
`net.ipv4.tcp_keepalive_time`) which require the kubelet to be
started with `--allowed-unsafe-sysctls=<glob,glob>`. Setting an
unsafe sysctl on a cluster that hasn't been configured for it
keeps the Pod Pending forever.

**When to use:** Use for connection-tuning sysctls (port range, keepalive, somaxconn)
on high-throughput network workloads. Confirm in advance whether
the target sysctl is on the safe list or requires kubelet
`--allowed-unsafe-sysctls`. For host-level sysctls (anything not
Pod-namespaced) — use a privileged DaemonSet, not Pod sysctls.

**Related fields:**
- `securityContext`
- `sysctls`

**Common mistakes:**
- Setting an unsafe sysctl on a cluster without `--allowed-unsafe-sysctls` — Pod stays Pending with `ForbiddenSysctl`; the failure is at admission time but easy to miss.
- Trying to set a host-level (non-Pod-namespaced) sysctl — silently ignored or rejected; these need to be applied node-side via DaemonSet or kubelet config.
- Quoting numeric values wrong — sysctl values are strings; `value: 60` parses as integer in YAML, which most kubelets reject. Always quote: `value: "60"`.

**Example:**

```yaml
name: net.ipv4.ip_local_port_range
value: 1024 65535
```

#### name

_Type:_ `string`

Name of a property to set

#### value

_Type:_ `string`

Value of a property to set

### k8s.io.api.core.v1.TCPSocketAction

- `readinessProbe`
- `tcpSocket`

- Probing a named port that isn't declared in `containers[].ports[]` — resolution fails and the probe is treated as failed every interval.

### k8s.io.api.core.v1.Toleration

Lets a Pod be scheduled onto (or remain on) a node that carries a
matching taint. Each entry pairs a `key` and `effect` with an
`operator` of either `Equal` (must also supply `value`) or `Exists`
(matches any value, and an empty `key` matches every taint). The
`effect` must match the node taint's effect exactly — `NoSchedule`,
`PreferNoSchedule`, or `NoExecute`; an empty `effect` matches all
three. `tolerationSeconds` only applies to `NoExecute` and controls
how long the Pod survives after the taint is added. The chart
exposes this list through `tolerations:` on Deployment and Job
workloads and passes it through unchanged.

**When to use:** Use to schedule onto tainted node pools (GPU, spot, dedicated tiers)
or to extend the eviction window for transient node conditions. If
you instead want to *prefer* certain nodes without requiring them,
use `nodeAffinity` (with the preferred variant) — tolerations only
remove a scheduling block, they don't attract pods.

**Related fields:**
- `tolerations`
- `nodeSelector`
- `affinity`

**Common mistakes:**
- Forgetting the `effect:` field with the intent of matching all effects but the cluster's taint only carries one — the toleration silently fails to match and the pod cannot schedule onto the tainted node.
- Using `operator: Equal` without populating `value:` — Kubernetes validates this and rejects the Pod; switch to `operator: Exists` if you want a value-agnostic match.
- Omitting `tolerationSeconds` for `effect: NoExecute` — the default is to tolerate forever, which usually defeats the point of the `NoExecute` eviction and keeps pods on misbehaving nodes.

**Example:**

```yaml
effect: NoSchedule
key: dedicated
operator: Equal
value: gpu
```

#### effect

_Type:_ `string`

Effect indicates the taint effect to match. Empty means match all taint effects. When specified, allowed values are NoSchedule, PreferNoSchedule and NoExecute.

#### key

_Type:_ `string`

Key is the taint key that the toleration applies to. Empty means match all taint keys. If the key is empty, operator must be Exists; this combination means to match all values and all keys.

#### operator

_Type:_ `string`

Operator represents a key's relationship to the value. Valid operators are Exists, Equal, Lt, and Gt. Defaults to Equal. Exists is equivalent to wildcard for value, so that a pod can tolerate all taints of a particular category. Lt and Gt perform numeric comparisons (requires feature gate TaintTolerationComparisonOperators).

#### tolerationSeconds

_Type:_ `integer`

TolerationSeconds represents the period of time the toleration (which must be of effect NoExecute, otherwise this field is ignored) tolerates the taint. By default, it is not set, which means tolerate the taint forever (do not evict). Zero and negative values will be treated as 0 (evict immediately) by the system.

#### value

_Type:_ `string`

Value is the taint value the toleration matches to. If the operator is Exists, the value should be empty, otherwise just a regular string.

### k8s.io.api.core.v1.TopologySpreadConstraint

Declarative rule that bounds the difference (`maxSkew`) in matching-Pod
count across a topology domain selected by `topologyKey` (typically
`topology.kubernetes.io/zone` or `kubernetes.io/hostname`). When the
scheduler would otherwise create a placement that violates the skew, it
either refuses (`whenUnsatisfiable: DoNotSchedule`, hard) or falls back
to the best placement available (`whenUnsatisfiable: ScheduleAnyway`,
soft). This is the modern, predictable replacement for weighted pod
anti-affinity for zone/host spreading; the chart's
`autoCreateSoftAntiAffinity` provides a coarse soft alternative when
you don't want to write a full constraint by hand.

**When to use:** Use whenever you need predictable spread across zones or hosts. Prefer
this over weighted `podAntiAffinity` — the scheduler reasons about
skew directly, instead of summing weights, so behavior is easier to
reason about. For "best effort" rough zone spread on a small chart
footprint, the chart's `autoCreateSoftAntiAffinity` is simpler.

**Related fields:**
- `topologySpreadConstraints`
- `affinity`
- `autoCreateSoftAntiAffinity`

**Common mistakes:**
- Defaulting `whenUnsatisfiable` to `ScheduleAnyway` when you actually want hard spread — the field's API default is `ScheduleAnyway`, so omitting it gives soft semantics; spelling out `DoNotSchedule` is required for a hard guarantee.
- Setting `maxSkew` to a number larger than `replicas` — the constraint is mathematically always satisfied and provides no spread, but reads like real protection on review.
- Pointing the `labelSelector` at labels the workload doesn't actually set (typos, wrong `app.kubernetes.io/instance`) — the constraint matches zero Pods, skew is always 0, and spread appears to work in tests but does nothing in production.

**Example:**

```yaml
labelSelector:
  matchLabels:
    app.kubernetes.io/name: my-app
maxSkew: 1
topologyKey: topology.kubernetes.io/zone
whenUnsatisfiable: DoNotSchedule
```

#### labelSelector

LabelSelector is used to find matching pods. Pods that match this label selector are counted to determine the number of pods in their corresponding topology domain.

#### matchLabelKeys

_Type:_ `array`

MatchLabelKeys is a set of pod label keys to select the pods over which spreading will be calculated. The keys are used to lookup values from the incoming pod labels, those key-value labels are ANDed with labelSelector to select the group of existing pods over which spreading will be calculated for the incoming pod. The same key is forbidden to exist in both MatchLabelKeys and LabelSelector. MatchLabelKeys cannot be set when LabelSelector isn't set. Keys that don't exist in the incoming pod labels will be ignored. A null or empty list means only match against labelSelector.

This is a beta field and requires the MatchLabelKeysInPodTopologySpread feature gate to be enabled (enabled by default).

#### maxSkew

_Type:_ `integer`

MaxSkew describes the degree to which pods may be unevenly distributed. When `whenUnsatisfiable=DoNotSchedule`, it is the maximum permitted difference between the number of matching pods in the target topology and the global minimum. The global minimum is the minimum number of matching pods in an eligible domain or zero if the number of eligible domains is less than MinDomains. For example, in a 3-zone cluster, MaxSkew is set to 1, and pods with the same labelSelector spread as 2/2/1: In this case, the global minimum is 1. | zone1 | zone2 | zone3 | |  P P  |  P P  |   P   | - if MaxSkew is 1, incoming pod can only be scheduled to zone3 to become 2/2/2; scheduling it onto zone1(zone2) would make the ActualSkew(3-1) on zone1(zone2) violate MaxSkew(1). - if MaxSkew is 2, incoming pod can be scheduled onto any zone. When `whenUnsatisfiable=ScheduleAnyway`, it is used to give higher precedence to topologies that satisfy it. It's a required field. Default value is 1 and 0 is not allowed.

#### minDomains

_Type:_ `integer`

MinDomains indicates a minimum number of eligible domains. When the number of eligible domains with matching topology keys is less than minDomains, Pod Topology Spread treats "global minimum" as 0, and then the calculation of Skew is performed. And when the number of eligible domains with matching topology keys equals or greater than minDomains, this value has no effect on scheduling. As a result, when the number of eligible domains is less than minDomains, scheduler won't schedule more than maxSkew Pods to those domains. If value is nil, the constraint behaves as if MinDomains is equal to 1. Valid values are integers greater than 0. When value is not nil, WhenUnsatisfiable must be DoNotSchedule.

For example, in a 3-zone cluster, MaxSkew is set to 2, MinDomains is set to 5 and pods with the same labelSelector spread as 2/2/2: | zone1 | zone2 | zone3 | |  P P  |  P P  |  P P  | The number of domains is less than 5(MinDomains), so "global minimum" is treated as 0. In this situation, new pod with the same labelSelector cannot be scheduled, because computed skew will be 3(3 - 0) if new Pod is scheduled to any of the three zones, it will violate MaxSkew.

#### nodeAffinityPolicy

_Type:_ `string`

NodeAffinityPolicy indicates how we will treat Pod's nodeAffinity/nodeSelector when calculating pod topology spread skew. Options are: - Honor: only nodes matching nodeAffinity/nodeSelector are included in the calculations. - Ignore: nodeAffinity/nodeSelector are ignored. All nodes are included in the calculations.

If this value is nil, the behavior is equivalent to the Honor policy.

#### nodeTaintsPolicy

_Type:_ `string`

NodeTaintsPolicy indicates how we will treat node taints when calculating pod topology spread skew. Options are: - Honor: nodes without taints, along with tainted nodes for which the incoming pod has a toleration, are included. - Ignore: node taints are ignored. All nodes are included.

If this value is nil, the behavior is equivalent to the Ignore policy.

#### topologyKey

_Type:_ `string`

TopologyKey is the key of node labels. Nodes that have a label with this key and identical values are considered to be in the same topology. We consider each <key, value> as a "bucket", and try to put balanced number of pods into each bucket. We define a domain as a particular instance of a topology. Also, we define an eligible domain as a domain whose nodes meet the requirements of nodeAffinityPolicy and nodeTaintsPolicy. e.g. If TopologyKey is "kubernetes.io/hostname", each Node is a domain of that topology. And, if TopologyKey is "topology.kubernetes.io/zone", each zone is a domain of that topology. It's a required field.

#### whenUnsatisfiable

_Type:_ `string`

WhenUnsatisfiable indicates how to deal with a pod if it doesn't satisfy the spread constraint. - DoNotSchedule (default) tells the scheduler not to schedule it. - ScheduleAnyway tells the scheduler to schedule the pod in any location,
  but giving higher precedence to topologies that would help reduce the
  skew.
A constraint is considered "Unsatisfiable" for an incoming pod if and only if every possible node assignment for that pod would violate "MaxSkew" on some topology. For example, in a 3-zone cluster, MaxSkew is set to 1, and pods with the same labelSelector spread as 3/1/1: | zone1 | zone2 | zone3 | | P P P |   P   |   P   | If WhenUnsatisfiable is set to DoNotSchedule, incoming pod can only be scheduled to zone2(zone3) to become 3/2/1(3/1/2) as ActualSkew(2-1) on zone2(zone3) satisfies MaxSkew(1). In other words, the cluster can still be imbalanced, but scheduler won't make it *more* imbalanced. It's a required field.

### k8s.io.api.core.v1.Volume

A single entry in the Pod's `volumes[]` list. Each volume pairs a
unique `name` with exactly one volume source — `configMap`, `secret`,
`emptyDir`, `persistentVolumeClaim`, `projected`, `hostPath`, and many
more. Containers reference the volume by `name` from
`volumeMounts[].name`; the name is the only link, so a typo on either
side produces a silent "volume not mounted" condition the Pod still
starts with. The chart surfaces this list verbatim through the
Deployment/Job `volumes:` field; the matching `volumeMounts:` lives on
each container.

**When to use:** Use `configMap` or `secret` for non-binary configuration/credentials;
use `emptyDir` for scratch space that lives with the Pod; use
`persistentVolumeClaim` for durable storage that survives Pod restarts.

sources, prefer a `projected` volume over multiple separate volumes.

**Related fields:**
- `volumes`
- `volumeMounts`
- `persistentVolumeClaim`

**Common mistakes:**
- Mismatched `name` between `volumes[].name` and `volumeMounts[].name` — Kubernetes does not error; the mount simply doesn't happen, and the app starts without its config or data.
- Mounting a `secret` or `configMap` with `subPath: <key>` to land it as a single file — the kubelet's atomic update mechanism is bypassed for subPath mounts, so updates to the Secret/ConfigMap never propagate into the Pod.
- Using `emptyDir` for data that must survive Pod rescheduling — emptyDir lives with the Pod's lifecycle; rescheduling to another node returns an empty directory.

#### podCertificate

Projects an auto-rotating credential bundle (private key and certificate chain) that the pod can use either as a TLS client or server.

Kubelet generates a private key and uses it to send a PodCertificateRequest to the named signer.  Once the signer approves the request and issues a certificate chain, Kubelet writes the key and certificate chain to the pod filesystem.  The pod does not start until certificates have been issued for each podCertificate projected volume source in its spec.

Kubelet will begin trying to rotate the certificate at the time indicated by the signer using the PodCertificateRequest.Status.BeginRefreshAt timestamp.

Kubelet can write a single file, indicated by the credentialBundlePath field, or separate files, indicated by the keyPath and certificateChainPath fields.

The credential bundle is a single file in PEM format.  The first PEM entry is the private key (in PKCS#8 format), and the remaining PEM entries are the certificate chain issued by the signer (typically, signers will return their certificate chain in leaf-to-root order).

If these coincide with a certificate rotation, it is possible that the private key and leaf certificate you read may not correspond to each other.  Your application will need to check for this condition, and re-read until they are consistent.

The named signer controls chooses the format of the certificate it issues; consult the signer implementation's documentation to learn how to use the certificates it issues.

### k8s.io.api.core.v1.WeightedPodAffinityTerm

### k8s.io.apimachinery.pkg.apis.meta.v1.FieldsV1

FieldsV1 stores a set of fields in a data structure like a Trie, in JSON format.

Each key is either a '.' representing the field itself, and will always map to an empty set, or a string representing a sub-field or item. The string will follow one of these four formats: 'f:<name>', where <name> is the name of a field in a struct, or key in a map 'v:<value>', where <value> is the exact json formatted value of a list item 'i:<index>', where <index> is position of a item in a list 'k:<keys>', where <keys> is a map of  a list item's key fields to their unique values If a key maps to an empty Fields value, the field that key represents is part of the set.

The exact format is defined in sigs.k8s.io/structured-merge-diff

### k8s.io.apimachinery.pkg.apis.meta.v1.LabelSelector

Label-based selector with two forms that AND together: `matchLabels`
is a `{key: value}` map (every entry must match), `matchExpressions`
is a list of LabelSelectorRequirement entries (set-based `In`/`NotIn`
with values, presence-based `Exists`/`DoesNotExist`). An empty
LabelSelector (no matchLabels and no matchExpressions) matches *all*
objects in scope — different from `null`, which matches none. This
shape is reused everywhere a Kubernetes resource picks Pods,
ConfigMaps, Services, or other API objects: Deployment `selector`,
Service `selector`, ServiceMonitor `selector`, NetworkPolicy
`podSelector`, and PodAffinity rules.

**When to use:** Prefer `matchLabels` for simple equality (most selectors). Use
`matchExpressions` when you need set operations (`In` with multiple
values, `NotIn`, presence checks). Be careful with the empty case:
`selector: {}` matches everything in scope; either populate it or
omit the field entirely.

**Related fields:**
- `matchLabels`
- `matchExpressions`
- `selector`

**Common mistakes:**
- Setting an empty LabelSelector (`{}`) and being surprised by overly broad matches — NetworkPolicy with `podSelector: {}` targets every Pod in the namespace, ServiceMonitor with `selector: {}` scrapes every Service.
- Changing Deployment `selector.matchLabels` on an existing Deployment — the field is immutable; the apply fails with a clear error, but the fix is to delete and recreate, which is disruptive.
- Forgetting that selectors are case-sensitive — `App: foo` and `app: foo` are different keys.

**Example:**

```yaml
matchLabels:
  app.kubernetes.io/name: my-app
  tier: backend
```

#### matchExpressions

_Type:_ `array`

matchExpressions is a list of label selector requirements. The requirements are ANDed.

#### matchLabels

_Type:_ `object`

matchLabels is a map of {key,value} pairs. A single {key,value} in the matchLabels map is equivalent to an element of matchExpressions, whose key field is "key", the operator is "In", and the values array contains only "value". The requirements are ANDed.

### k8s.io.apimachinery.pkg.apis.meta.v1.LabelSelectorRequirement

One set-based selector entry. `key` is the label name, `operator`
is `In`, `NotIn`, `Exists`, or `DoesNotExist`, and `values` is a
list whose meaning depends on the operator. With `In`/`NotIn`,
`values` must be non-empty; with `Exists`/`DoesNotExist`, `values`
must be empty. Multiple LabelSelectorRequirement entries inside a
LabelSelector's `matchExpressions` AND together — and they also AND
with `matchLabels`.

**When to use:** Use inside `matchExpressions` when set semantics are needed: matching
one of several values (`In`), excluding a set (`NotIn`), or just
checking label presence (`Exists`/`DoesNotExist`). For plain `key:

**Related fields:**
- `matchExpressions`
- `matchLabels`
- `selector`

**Common mistakes:**
- Using `In` with an empty values list (or `Exists` with values) — the API validates this and rejects the parent resource.
- Forgetting that `NotIn` matches objects that *also* have no value for the label — set-theoretic 'not in {x,y}' includes 'absent'; pair with `Exists` if you want 'has the label but not these values'.

_Type:_ `string`

operator represents a key's relationship to a set of values. Valid operators are In, NotIn, Exists and DoesNotExist.

#### values

_Type:_ `array`

values is an array of string values. If the operator is In or NotIn, the values array must be non-empty. If the operator is Exists or DoesNotExist, the values array must be empty. This array is replaced during a strategic merge patch.

#### deletionTimestamp

DeletionTimestamp is RFC 3339 date and time at which this resource will be deleted. This field is set by the server when a graceful deletion is requested by the user, and is not directly settable by a client. The resource is expected to be deleted (no longer visible from resource lists, and not reachable by name) after the time in this field, once the finalizers list is empty. As long as the finalizers list contains items, deletion is blocked. Once the deletionTimestamp is set, this value may not be unset or be set further into the future, although it may be shortened or the resource may be deleted prior to this time. For example, a user may request that a pod is deleted in 30 seconds. The Kubelet will react by sending a graceful termination signal to the containers in the pod. After that 30 seconds, the Kubelet will send a hard termination signal (SIGKILL) to the container and after cleanup, remove the pod from the API. In the presence of network partitions, this object may still exist after this timestamp, until an administrator or automated process can determine the resource is fully terminated. If not set, graceful deletion of the object has not been requested.

Populated by the system when a graceful deletion is requested. Read-only. More info: https://git.k8s.io/community/contributors/devel/sig-architecture/api-conventions.md#metadata

#### finalizers

_Type:_ `array`

Must be empty before the object is deleted from the registry. Each entry is an identifier for the responsible component that will remove the entry from the list. If the deletionTimestamp of the object is non-nil, entries in this list can only be removed. Finalizers may be processed and removed in any order.  Order is NOT enforced because it introduces significant risk of stuck finalizers. finalizers is a shared field, any actor with permission can reorder it. If the finalizer list is processed in order, then this can lead to a situation in which the component responsible for the first finalizer in the list is waiting for a signal (field value, external system, or other) produced by a component responsible for a finalizer later in the list, resulting in a deadlock. Without enforced ordering finalizers are free to order amongst themselves and are not vulnerable to ordering changes in the list.

### k8s.io.apimachinery.pkg.apis.meta.v1.OwnerReference

Server-populated link from a child object to a parent — drives
Kubernetes garbage collection. When all owners are deleted, the
child is garbage-collected (unless `blockOwnerDeletion: true` is set
on the reference, in which case the parent stays in Terminating
until the child is gone). `controller: true` marks one owner as the
*managing* controller — only one such entry per child; field
managers and HPA scale targets typically check it. The chart never
writes OwnerReferences; Kubernetes adds them automatically when a
controller (Deployment → ReplicaSet → Pod) creates child resources.

**When to use:** You rarely write this in manifests. When implementing a controller,
set owner references on objects you create so they're garbage-
collected when the parent CR is deleted. Use `controller: true` on
exactly one reference and `blockOwnerDeletion: true` if the parent's
finalization needs to wait for children.

**Related fields:**
- `metadata`
- `ownerReferences`
- `finalizers`

**Common mistakes:**
- Setting `controller: true` on multiple references — only one controller per object is allowed; the API rejects the second.
- Forgetting `uid` — the field is required; without it the API rejects the reference. Pull the UID from the parent's `metadata.uid` at create time.
- Cross-namespace ownerReferences — not supported; garbage collection only works within a single namespace (cluster-scoped owners are an exception). The reference is silently ignored.

**Example:**

```yaml
apiVersion: apps/v1
blockOwnerDeletion: true
controller: true
kind: Deployment
name: my-app
uid: 12345678-1234-1234-1234-123456789abc
```

#### apiVersion

_Type:_ `string`

API version of the referent.

#### blockOwnerDeletion

_Type:_ `boolean`

If true, AND if the owner has the "foregroundDeletion" finalizer, then the owner cannot be deleted from the key-value store until this reference is removed. See https://kubernetes.io/docs/concepts/architecture/garbage-collection/#foreground-deletion for how the garbage collector interacts with this field and enforces the foreground deletion. Defaults to false. To set this field, a user needs "delete" permission of the owner, otherwise 422 (Unprocessable Entity) will be returned.

#### controller

_Type:_ `boolean`

If true, this reference points to the managing controller.
